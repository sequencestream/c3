## 基本要求

- 文档第一读者是人类, 第二读者才是Agent，优先使用分层和结构化的表述
- 文档要简洁明了
- 保持文档和代码的同步更新

## 宪法级要求 (不可违背)

- **只描述现状, 不记录历史**: 文档正文只描述系统"当前是什么样", 用陈述句写终态。禁止出现"新增了"、"原来是 A 改为 B"、"已废弃"、"临时方案"、日期/版本/PR 号等变更叙述。变更历史属于 git commit、ADR (决策)、database/migrate (表结构), 不进规格正文。
- **每次修改都是一次重构**: 同步代码时不允许只做加法。改动一处, 就要连带删除/合并该主题下过时、重复、矛盾的旧内容, 使这一段落读起来像是一次性写成的整体, 而非层层叠加的补丁。修改后该主题的文字总量应尽量持平或减少。
- **单一事实源, 不复制**: 同一事实只在一处权威文档描述, 其余位置用引用/链接指向, 禁止跨文档粘贴同样的内容。发现重复即合并到权威源。
- **删除过时内容无需征求同意**: 与代码不符、已无对应实现、被新内容取代的段落, 直接删除或改写, 不必先询问。
- **每行都要挣得存在**: 删掉套话、显而易见的废话、为凑篇幅的解释。若一句话去掉后信息无损, 就去掉。宁缺毋滥。
- **顺带清理是义务**: 无论本次任务是否要求, 只要在编辑中读到附近有违反上述原则的内容, 顺手修正, 不留待"以后"。

## 格式

- 表格字段内容不易过长, 如果内容过长, 则不使用表格，改用段落描述，list 格式。
