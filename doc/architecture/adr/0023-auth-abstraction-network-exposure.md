# 0023 — 认证抽象边界（basic 为首个 provider）

- **Status:** proposed（「网络暴露的强制前提」框架已退役，见下）
- **Date:** 2026-06-11
- **Driver:** 先立可扩展的认证抽象边界，而非把单一认证方式焊死在各层

> **更新（C-SEC-5 前提退役）：** 本 ADR 原以「server 仅绑定 localhost、认证是网络暴露的强制前提」
> 为立论背景。该前提已退役:**是否把 c3 暴露到网络、以及绑定到哪个地址,由使用者决定**,认证不再是
> 非回环绑定的强制前置;宪法 C-SEC-5 条款已随之移除。仍然有效的是**认证抽象本身**的决策
> (`none`/`basic` provider、会话令牌策略、唯一管理员门等 provider 无关部分),记录在
> [auth 域](../../domains/core/auth/auth-overview.md)。下文中出现的「localhost-only / 非回环需认证」
> 措辞按此退役理解,不再作为约束。

> **更新（后续废弃）：** 曾规划的 `oauth`/OIDC provider 已移除。认证抽象当前只支持 `none`
> 与 `basic` 两种 provider；相关的 OAuth 契约字段、保存校验与「OAuth 运行时（延后）」设计不再适用。
> 本 ADR 其余关于 `bindAddress`、会话令牌策略、唯一管理员门等 provider 无关的决策保持不变。
> 抽象仍保留「新增一个 `kind` 分支即扩展」的扩展点，供未来 SSO/多用户使用。

## Context

当前 Web UI 完全没有任何认证。这在仅本机使用时可接受,但一旦把 server 暴露到非回环地址,任何能访问该网卡的人都能驱动 agent 执行工具、读写文件、动用宿主 CLI 凭据。是否暴露由使用者决定;认证是面向暴露场景的一项**能力**,用于在暴露时先确立连接身份。

这是「网络暴露」四件套的**第一件（地基）**，后续三件依赖本件确立的边界：

1. 本件——认证抽象边界 + 契约（无运行时）。
2. 口令哈希/令牌签名等运行时实现 + 认证中间件。
3. 登录页 + 会话生命周期 UI。
4. 设置持久化文件的权限收紧与日志脱敏。

### 落地进度（2026-06-11，runtime 切片）

第 2、3 件的**前半**已落地——为系统设置的 basic 认证配置面板提供端到端可用的最小运行时：

- **口令哈希**：认证特性的口令模块，基于 `node:crypto` 的 `scrypt` 的零依赖 PHC 串
  `$scrypt$…`（兼容 `bun build --compile`）；明文永不落盘，校验为常数时间。
- **login 真实校验**：login 在持久化 basic provider 的账号集内按用户名查找并验口令（**任一账号**
  皆可登录——管理员是改配置的权限来源，非登录特权，本期无 RBAC），成功签发**不透明随机** token + TTL 派生
  过期时刻（token **签名/验签**仍延后）。
- **账号管理消息（2026-06-16 多账号演进）**：set_admin_password（携 username/password/可选 currentPassword）升级为
  **upsert 账号口令**（新用户名 ⇒ 新增账号，首账号自动成为管理员；已存在 ⇒ 改密）；新增 remove_account
  与 set_admin_account 管理账号集与唯一管理员指派。明文仅传输期存在、服务端哈希后写入。敏感操作门闩
  =「证明当前口令」：改已存在账号须带其当前口令（currentPassword），新增账号免验（引导阶段信任本地操作者）。删管理员
  且尚有其它账号 ⇒ 拒（`admin_must_reassign`，须先改管理员）；删唯一账号 ⇒ 清空回未配置态。
- **保存设置不碰 basic 凭据**：basic 账号集（用户名/哈希/管理员指派）由上述三消息独占；保存设置
  持久化时强制用磁盘 basic provider **整体覆盖**（保存层的「保留磁盘 basic provider」步骤），杜绝陈旧/空草稿回写覆盖、改管理员或
  抹账号。basic provider 的启用态派生：账号集非空且管理员用户名有效 ⇔ 启用。
- **配置面板**：设置面板的认证区——provider 二态下拉（**无需认证 / basic**，均可选）、
  basic **账号列表编辑器**（每行：管理员单选 radio + 改密入口 + 删除；底部新增账号行，均 write-only 不回显哈希）、
  网络暴露开关；**未配管理员**时暴露开关置灰。下拉即认证总开关（已无独立
  「启用认证」复选框）：选「无需认证」⇒ provider 种类为 none、启用态为 false、免登录；选 basic ⇒ 配好管理员
  后启用态派生为 true。账号增删改/设管理员各走
  专用消息即时持久化（结果 toast + 刷新设置），不依赖保存按钮。

**仍延后**（第 2、3、4 件未尽部分）：token 签名/验签；WS 握手与请求级**认证中间件强制**；把
`exposure.bindAddress` 接入 server 实际绑定的运行时应用（本切片**不**改变实际绑定）；
会话生命周期完整 UI；设置文件权限收紧与日志脱敏。会话令牌与签名密钥引用仍为待用契约。

设计这层抽象时面临的核心张力：**只需要 basic 登录，却不能只为 basic 建模。** 若把「用户名 + 口令」直接铺在系统设置顶层、把登录消息写死成只含用户名与口令两字段的固定形状，则将来加 SSO/多用户时，配置层、协议层、校验层、UI 层都要返工——而这正是可扩展认证抽象应当避免的。

约束：

- 共享协议定义是类型唯一源、**零运行时**（ADR-0009）；任何校验/哈希/签名实现都不得进入 wire 模块。
- 系统设置持久化于 `~/.c3/settings.json`，新增字段必须**向后兼容**：缺省即「未启用认证」，旧文件行为完全不变。
- 设置文件将开始承载**口令哈希**（不是明文）——这是被接受的成本，文件权限与脱敏由后续件保证。
- 不在本件实现任何运行时逻辑（中间件、登录页、哈希函数、令牌签名/校验均不在内）。

## Options Considered

### 1. 不立抽象，直接铺 basic 字段

在系统设置顶层加用户名 / 口令哈希 / 认证开关三个扁平字段，登录消息写成只含类型、用户名、口令的固定形状。

_Pro:_ 最少的代码，直达 basic 需求。
_Con:_ 单一认证方式焊死在配置层与协议层——加 SSO/多用户时四层返工。
_Con:_ 没有「provider」概念，无法表达「同一套会话令牌模型 + 多种认证后端」。
_Con:_ 与既有的 vendor-neutral（ADR-0011）、双层配置（ADR-0021）等「先立边界再填实现」的工程范式不一致。

### 2. 完整多用户 + RBAC + 多 provider 一次到位

直接建模用户表、角色、权限、SSO/LDAP 多 provider。

_Pro:_ 一步到位，未来不返工。
_Con:_ 当前没有多租户/组织概念——严重超前设计（与 ADR-0021 否决三层配置同理）。
_Con:_ 巨大的未验证表面积；与「本期只需 basic」的实际需求脱节。
_Con:_ 无数据库（架构约束：no database），多用户持久化无处安放。

### 3. 可扩展抽象 + 单一 basic provider 实现（selected）

把认证 provider 建模为按种类区分的判别式联合（none 与 basic），会话令牌模型与登录/登出/未认证消息**与 provider 无关**（对任何 provider 复用），认证配置作为系统设置的可选字段挂入。SSO/多用户仅在联合上「留位」——新增一个种类即扩展，不动既有分支、不动会话令牌模型、不动消息契约。

_Pro:_ 边界与实现解耦——本期只落 basic，将来加 provider 只追加一个联合分支。
_Pro:_ 会话令牌/登录消息是 provider 无关的中立层，跨 provider 复用（与 ADR-0011 的 vendor 中立同构）。
_Pro:_ 启用主开关 + 可选认证配置——缺省即未启用，旧配置零行为变化（与 ADR-0021 的「启用」范式一致）。
_Pro:_ 口令以**哈希**（PHC 串）存储、签名密钥以**引用**（env 名/keystore id）而非本体存储——把「设置文件含敏感物」的暴露面降到最小。
_Con:_ 比选项 1 多一层联合/嵌套结构；本期只有一个分支，抽象「暂时空转」——但这正是地基件的目的。

### 4. 单层扁平认证配置（有启用开关但无 provider 联合）

认证配置为扁平结构（启用开关、用户名、口令哈希、TTL 等），无 provider 种类区分。

_Pro:_ 比选项 3 略简单。
_Con:_ 仍把 basic 的字段（用户名/口令哈希）焊在顶层；加 SSO/多用户时这些字段无意义，要么留空污染、要么再返工拆出 provider——回到选项 1 的问题。

## Decision

**采纳选项 3。** 建立一个可扩展的认证抽象：认证是面向网络暴露场景的一项能力，暴露时建议先启用认证以确立连接身份。本期只确立**边界与契约**，运行时实现留待后续件。

### 类型结构（落共享协议定义，零运行时）

下述类型在共享协议定义中以零运行时形式声明，作为两端编译期一致的唯一来源。

**provider 种类（扩展点）**：认证方式枚举为两种——`none`（无认证，默认）、`basic`（运行时已上线）；sso/多用户后续追加。

**none provider**：无认证（免登录），是无认证默认的一等表达。无任何配置字段，种类本身即全部形状。不变量：种类为 none ⇔ 认证未启用（由规整步骤强制），故下拉「无需认证」与启用主开关永不打架（单一真相源，UI 只读 provider 种类）。

**basic provider（多账号）**：账号集 + 唯一管理员（2026-06-16 由单账号演进）。每个账号都可登录；管理员用户名指向其中之一（空集时为空串 = 未配置态），是改系统配置的权限来源（无 RBAC）。账号凭据仅由 set_admin_password（upsert）/remove_account/set_admin_account 三消息变更，保存设置不碰。账号模型：

- 用户名：trim 后大小写敏感唯一。
- 口令哈希：PHC 串（算法 + 参数 + 盐 + 摘要），如 `$scrypt$...`；绝不存明文。
- 账号集为 0..n 个账号；空 ⇒ 未配置（启用态派生为 false）。
- 管理员用户名：唯一管理员，必指向账号之一；空集时为空串。

**provider 抽象**：以上两种 provider（`none`、`basic`）构成按种类区分的判别式联合。

**会话令牌策略**：TTL + 签名密钥**引用**（env 名/keystore id），密钥本体绝不入设置文件。

**网络暴露/绑定意向**：可选绑定地址，缺省 `127.0.0.1`；如 `0.0.0.0` 表网络暴露意向，是否暴露由使用者决定。绑定地址的运行时应用由后续件接入。

**认证总配置**：含启用开关、provider、会话令牌策略、可选暴露配置；作为系统设置的可选字段挂入，缺省/未启用 ⇒ 未启用（默认）。

**会话令牌模型（provider 无关的中立层）**：含不透明令牌 id（jti）、主体（basic 下即用户名）、签发时刻与过期时刻（Unix ms，过期 = 签发 + TTL×1000）。

**登录/登出/未认证消息（provider 无关）**：HTTP 端点与 WS 共用同一契约。登录请求含用户名与口令（明文仅在传输期，校验后即弃）。登录失败码枚举为 `invalid_credentials`、`auth_disabled`、`rate_limited`。登录结果为成功（携 token + 过期时刻）或失败（携失败码）的二选一。

**WS 协议增补**（作为 wire 契约记录）：

| 方向            | 消息                    | 字段 / 语义                                                       |
| --------------- | ----------------------- | ----------------------------------------------------------------- |
| 客户端 → 服务端 | `login`                 | 携登录请求（用户名 + 口令）                                       |
| 客户端 → 服务端 | `logout`                | 无                                                                |
| 客户端 → 服务端 | `set_admin_password`    | username + password + 可选 currentPassword；upsert 账号口令       |
| 客户端 → 服务端 | `remove_account`        | username                                                          |
| 客户端 → 服务端 | `set_admin_account`     | username                                                          |
| 服务端 → 客户端 | `login_result`          | 携登录结果                                                        |
| 服务端 → 客户端 | `admin_password_result` | 携改密结果                                                        |
| 服务端 → 客户端 | `account_op_result`     | 结果码：成功 \| `not_found` \| `admin_must_reassign` \| `invalid` |
| 服务端 → 客户端 | `unauthenticated`       | reason：`missing` \| `expired` \| `invalid`（401 语义）           |

系统设置增补可选的认证配置字段，缺省 = 未启用，向后兼容。

### 不变量

1. **缺省即未启用**：系统设置的认证字段缺失，或认证未启用，或 provider 校验失败 ⇒ 等价于「无认证」（默认）。配置规整 fail-soft：非法认证配置被丢弃为缺省，而非抛错。
2. **`none` ⇔ 未启用（单一真相源）**：provider 种类为 none 与认证未启用语义重叠，约定二者等价，由配置规整步骤强制（none provider 携带「已启用」会被规整回未启用）。UI 只读 provider 种类、不另设第二开关，故「无需认证」下拉项与主开关永不打架。
3. **永不明文**：类型与示例中绝不出现口令明文。basic 账号的口令哈希是 PHC 串；登录请求的口令是仅存在于「传输期」的明文，校验后即弃、绝不持久化。
4. **引用而非本体**：签名密钥引用是签名密钥的**引用**（env 名/keystore id），本体绝不进 `settings.json`。校验层默认剥离未声明键，客户端误带的明文机密不会被持久化。
5. **provider 无关的中立层**：会话令牌、登录请求/结果、login/logout/unauthenticated 消息都不含 provider 专有字段，新增 provider 不改动它们。
6. **扩展点单点**：新增认证方式 = 给 provider 种类枚举加一个值 + 给 provider 联合追加一个分支 + 给服务端校验注册表追加一个分支；既有分支、会话令牌模型、消息契约均不动（与 ADR-0011 vendor 扩展、ADR-0021 双层配置同构）。
7. **唯一管理员引用完整（2026-06-16）**：单 provider 联合天然保证同一时刻只有一种认证方式启用。basic 下账号用户名唯一、非空时管理员用户名必指向其一。两层守：**保存层**对 UI 操作回结构化码（`account_op_result`），配置规整对手改磁盘 fail-soft 兜底——basic 悬空/重名管理员 ⇒ 整块丢弃为「无认证」。basic 启用态派生 ⇔ 账号集非空且管理员用户名有效。旧单账号（用户名 + 口令哈希）一次性迁移为「账号集 + 管理员用户名」（判据 = 无账号集字段，幂等）。

### 运行时校验位置

遵循 ADR-0009/ADR-0011 的分工：**类型**在共享协议定义（零运行时），**运行时校验 schema**在配置校验模块，底部以双向类型断言把 schema 与 wire 类型钉死防止漂移（复刻 agent 配置校验 schema 的体例）。配置规整步骤在加载时校验认证配置，非法 ⇒ 省略（fail-soft）。

## Consequences

### 正面

- **边界先于实现**：本期零运行时即固定跨层契约；后续三件在稳定边界上填实现，不返工。
- **可扩展**：SSO/多用户加一个联合分支即可，会话令牌与消息契约不动。`none` 与 `basic` 两种 provider 已按此扩展点落地，印证边界有效。
- **向后兼容**：旧 `settings.json` 无认证字段 ⇒ 行为完全不变（未启用 = 无认证现状）。
- **暴露面最小化**：口令存哈希、密钥存引用——设置文件即便泄露也不含明文口令或签名密钥本体。

### 负面 / 接受的成本

- 设置持久化文件从本件起**承载口令哈希**。哈希不可逆，但仍是敏感物——文件权限收紧（如 `chmod 600`）与日志脱敏**不在本件**，由后续件（第 4 件）保证。本 ADR 显式记录该未了责任。
- 抽象本期「空转」：只有一个 basic 分支，多出的联合/嵌套层在加入第二个 provider 前不产生直接收益——这是地基件的固有代价（与 ADR-0021 双层配置在单项目时的空转同理）。
- **绑定地址尚未接入运行时**：本件只立契约,`exposure.bindAddress` 尚未接入 server 的实际绑定,由后续件接入。是否暴露、绑定到哪个地址由使用者决定;暴露到网络时建议启用认证。

## Compliance

- 配置校验 schema 必须有底部双向类型钉死（schema 推断类型 ↔ 认证配置类型），任一方向失败即编译错误。
- 配置规整对以下输入必须产出「未启用」：认证字段缺失、未启用、provider 校验失败、未知 provider 种类。旧 `settings.json`（无认证字段）round-trip 后行为不变。
- 配置校验测试必须覆盖：合法 basic 配置解析通过、缺省 ⇒ 缺省（未启用）、非法（未知种类 / 缺字段 / 明文位置）⇒ 拒绝。
- 协议测试必须含 login/logout/login_result/unauthenticated 消息的 JSON round-trip。
- 类型与测试中**绝不**出现真实口令明文常量（测试用占位口令仅作传输期入参，不作持久化值）。
- 类型检查必须通过。
