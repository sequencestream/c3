# Constitution — c3

项目的最高层约束。这些约束极少变化,并且优先于任何规格、设计与代码。
对本文档的修改需要维护者的明确签署(见修订流程)。

## 使命与价值观

**使命:** 提供一个统一的 AI 工作台,集中管理并驱动多个 AI 智能体的工作,构建一个安全、可控、可信的闭环。

**价值观,按优先级排序**(冲突时优先级高者胜出):

1. **决策边界的安全性** — 没有经过用户授权的决策,任何智能体的敏感工具都不得运行。这一条压倒一切。
2. **通信契约的正确性** — 浏览器与服务端在任何时刻都对消息形状达成一致。协议损坏或含糊比缺失某个功能更糟。
3. **人对多智能体的掌控** — 人始终是工作的编排者与最终决策者。并发运行多个智能体是常态,但每个智能体都在用户可见、可干预的控制下运行;绝不把控制权悄悄让渡给自动化。
4. **简洁性优先** — 以本地进程运行、不依赖数据库。优先选择简单直接的设计,而非提前引入多租户、分布式或云端通用性带来的复杂度。
5. **开发者体验** — 可读的工具输入、快速的反馈、易于构建与自托管。

**冲突裁决规则:** 当两个价值观冲突时,满足优先级更高的一个,并在相应的 ADR 中记录取舍。

## 边界基线

c3 是一个 AI 工作台。以下边界定义了它"是什么"与"绝不成为什么"。

**未经 ADR 禁止跨越的边界:**

- 在未经用户明确授权的情况下,把用户的代码、凭据或工作数据送往本机之外(云同步、遥测、第三方分析);
- 任何绕过或削弱浏览器决策网关(见安全基线)的旁路。

**明确属于工作台范围内、无需再论证的能力:** 编排多个 vendor 的智能体;本地持久化会话、意图、
讨论、自动化等工作数据;并发运行多个会话与工作区。

**例外流程:** 跨越上述边界需要在 `architecture/adr/` 下新建一个 ADR,列出所考虑的方案,并由维护者批准。

## 安全基线(不可协商)

- **C-SEC-1 — 权限网关。** c3 是它所编排的每一个智能体会话的权限网关。会话继承宿主与项目配置
  (各 vendor 的 `~/.claude`、项目 `.claude` 等)中的 hook 与允许/拒绝规则,这些规则优先生效;
  任何未被它们预先决定的敏感工具调用都必须流经 c3 的决策通道并呈现给用户。继承的允许规则可能自动
  批准某个浏览器从未见过的工具——这是可接受的,与直接使用对应 vendor CLI 的行为一致。改变这一继承
  模型需要新建一个 ADR。
- **C-SEC-2 — 授权前置。** 被判定为敏感的工具,除非有明确的决策授权——用户的允许(Allow),或一个
  授权自动执行的活跃权限模式——否则不得执行。
- **C-SEC-3 — 默认拒绝。** 在没有任何决策的情况下,默认结果是**拒绝**。待决请求无限期阻塞,直到用户
  做出决定(无超时);运行被中止则结果视为拒绝。无法解析或未知的客户端消息会被忽略,绝不当作批准处理。
- **C-SEC-4 — 不碰凭据。** 不硬编码也不记录任何密钥。各 vendor CLI 自己拥有认证权;c3 从不处理智能体
  供应方的凭据。

## 核心原则

**工程:**

- 处处使用 TypeScript `strict`。用可辨识联合类型(discriminated union)建模通信与状态,基于 `type`
  做窄化;边界处使用 `unknown` 并校验外部输入。禁止用 `as` 漂白类型。
- 通信协议只有唯一真源(`@ccc/shared`)。两端导入同一份定义,谁都不重新定义。
- **vendor 中立** — 面向统一的智能体抽象编程,而非某一家 CLI 的细节;新增一个 vendor 不应牵动核心编排。

**多智能体编排:**

- 智能体在用户控制的权限模式下运行。升级到更宽松的模式(如跳过权限确认)只能通过一次明确的、可观察的
  UI 操作完成,绝不能悄悄进行。
- 会话内是串行的:对一个正在进行中的会话发起新的 prompt 会被拒绝,而不是被合并。不同会话并发运行,没有
  固定上限。运行归属于进程级别的会话运行时,而非某个连接;切换正在查看的会话或关闭 socket 都不会停止
  运行——只有明确的停止/删除操作才会。

**运维:**

- 交付为单一自包含二进制,可自托管、离线运行核心功能。
- 找不到某个 vendor 的可执行文件,或其运行时出错,都要以明确的错误呈现给用户,绝不默默挂起。

## 文档撰写规范

- 文档在边界高度解释设计——实现方式、流程、逻辑、状态与规则——清晰到足以据此评审和实现。避免穷举式地
  列出与源码重复、且会随之漂移失步的低层级细节(完整源码树、逐文件或逐符号清单);共享契约只记录一次,
  并通过引用复用。

## 修订流程

由维护者提出并签署变更。放宽某条 `C-SEC-*` 规则或跨越边界基线的修订,需要在 ADR 中给出书面理由,并在
变更记录中做出说明。违反本文档的行为将被视为阻断发布的缺陷。
