# sandbox — Domain Spec

## Overview

The sandbox domain provides **container-based isolation** for agent execution. Rather than running vendor CLI processes directly on the host, the sandbox layer creates lightweight containers with configurable resource limits, filesystem isolation, network controls, and seccomp security profiles.

This is a **kernel infrastructure** domain (inner domain, ADR-0009) — it provides the plumbing that higher-level features (runs, schedules, permissions) consume through a single sandbox-driver capability contract. It knows nothing about agents, sessions, or business logic.

**Scope:** the sandbox configuration model, the driver capability contract, the configuration schema and merge rules, the named-definition registry, the Docker-backed runtime, and seccomp profile loading/merging.
**Boundary:** it does not manage containers directly (the driver does), does not decide which sandbox to use (the caller does), and does not implement container orchestration (Kubernetes, Swarm).

## Capabilities

The domain is composed of five capabilities:

- **Configuration model** — the value shapes that describe a sandbox: its type, a system-supplied definition, the per-workspace overrides layered on top, the fully resolved configuration, a live container handle, and the result / health / option shapes the driver returns or accepts.
- **Driver contract** — a single capability contract every runtime backend implements (start, stop, exec, stream, snapshot, health check; see § Driver capability contract).
- **Configuration schema + merge** — validates a system definition and a workspace config, then merges them into a resolved configuration (system base + workspace overrides, deny-by-default security policies; see SND-R3 / R4 / R5 / R24).
- **Named-definition registry** — stores system definitions by name and resolves a name plus workspace overrides into a runnable configuration (register / get / resolve / membership test / list names / count).
- **Docker runtime + seccomp** — the Docker-backed implementation of the driver contract, plus loading the default seccomp profile and merging an override profile over a base.

### Layer Architecture

Higher-level features (schedules, runs) call the sandbox driver only through its capability contract. The kernel sandbox layer holds the driver contract (implemented by the Docker-backed runtime), the registry (which registers system definitions and resolves them against workspace overrides), and the configuration parser/validator. The Docker runtime talks to the local Docker daemon; the seccomp loader reads its profiles from packaged JSON.

### Capability responsibilities

| Capability                   | Responsibility                                                                                                                                                                    |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Configuration model          | The value shapes: sandbox type, system definition, workspace overrides, resolved configuration, live container handle, exec result, health status, and start/stop option bundles. |
| Driver contract              | The capability contract with the six operations: start / stop / exec / stream / snapshot / health check (see § Driver capability contract).                                       |
| Configuration schema + merge | Validates a system definition and a workspace config, then merges them into a resolved configuration.                                                                             |
| Named-definition registry    | Stores system definitions by name; offers register, get, resolve (name + workspace overrides → runnable config), membership test, list names, and count.                          |
| Docker runtime               | The Docker-backed implementation of the driver contract.                                                                                                                          |
| Default seccomp profile      | A Docker-compatible, MVP-permissive seccomp profile (allow-all action).                                                                                                           |
| Seccomp loading + merge      | Load the default profile, load a profile from a path, and merge an override profile over a base.                                                                                  |

### Driver capability contract

Every runtime backend implements the same six operations. Behaviour is described as input → output; the Docker column states how the Docker-backed runtime realises each.

| Operation    | Input                                           | Output                      | Docker realisation                                          |
| ------------ | ----------------------------------------------- | --------------------------- | ----------------------------------------------------------- |
| start        | resolved configuration + optional start options | a live container handle     | create the container, then start it                         |
| stop         | a container handle + optional stop options      | nothing                     | stop the container, optionally remove it                    |
| exec         | a container handle + an argv vector             | exit code + captured output | run the argv inside the container and inspect the result    |
| stream       | a container handle + an argv vector             | a readable byte stream      | run the argv inside the container and return its raw stream |
| snapshot     | a container handle + a tag                      | a snapshot image id         | commit the running container to an image                    |
| health check | a container handle                              | a health status             | inspect the container's running state                       |

## Business Rules

| ID      | Rule                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| SND-R1  | A sandbox must be **Docker runtime only** in Phase 1 (MVP). gVisor, Kata, and Firecracker are Phase 2 (planned).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| SND-R2  | Each sandbox definition has a unique name. The registry overwrites on re-registration.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| SND-R3  | A workspace config references a system definition by name. Resolving that reference merges the system definition with the workspace overrides.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| SND-R4  | Workspace overrides take precedence over system-definition values for the same field. Environment variables are **merged** (not replaced), with workspace values winning on conflict.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| SND-R5  | Unspecified optional fields get sensible defaults: a memory limit of 512 MB, a CPU limit of one core, `networkDisabled` on, `readonlyRootfs` on, and an empty environment-variable map. The two security policies (`networkDisabled`, `readonlyRootfs`) are **deny-by-default** and resolved from the **workspace** config (SND-R24), not the system definition.                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| SND-R10 | Sandboxed containers run with **`--network none`** by default (`networkDisabled: true`, a **workspace** policy — SND-R24). Network egress requires either flipping the workspace `networkDisabled` to `false` or explicit allowlist configuration (Phase 2).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| SND-R11 | The `networkAllowlist` configuration is the extension point for egress rules. Supplying a non-empty list is **rejected** in MVP (Phase 2 will introduce MITM proxy filtering).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| SND-R12 | Resource limits can be set via the flat memory-limit / CPU-limit fields OR via the structured resource-limits sub-object. When both are set, the structured values **take precedence**. The structured stop-timeout field is the only way to set the container stop timeout.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| SND-R6  | The Docker runtime connects to the **local Docker daemon** via the default socket. Remote Docker hosts are not supported (constitution: localhost-only).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| SND-R7  | Stopping a sandbox swallows errors from already-stopped containers (idempotent).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| SND-R8  | A health check reports an error status with the container not running (never raising) when inspection fails.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| SND-R9  | The default seccomp profile is permissive (allow-all action) for MVP. Hardening will follow in Phase 2.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| SND-R13 | Sandbox containers serve **only the worktree intent-dev run** — a run whose effective working directory is an isolated worktree. A plain chat run (no isolated working directory) and a current-branch dev run (whose sandbox config the worktree-only normalisation strips) never sandbox (ADR-0024).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| SND-R14 | Launching a sandbox keys the sandbox **config** by the workspace identity and bind-mounts the **worktree** (the run's effective working directory) at `/workspace`. It also derives that workspace's centralized specs root and bind-mounts it read/write at the **same absolute path** in the container. Thus an absolute spec path in a dev prompt resolves identically on host and in sandbox; the root is derived from the owning workspace, never the worktree. Container labels carry both the workspace (`c3.project`) and the mounted worktree (`c3.worktree`) (ADR-0024).                                                                                                                                                                                                                        |
| SND-R15 | When sandbox is enabled, exactly **one custom agent is picked at random** from the normalised agent pool and **pinned** onto the (pending) run; its vendor decides the container binary and its provider connection flows into the container. Supported picks: **claude** (provider env carried via the claude wrapper), **codex DIRECT** (base URL and model ride the wrapper's forwarded argv; the API key is mirrored into the env-file as `CODEX_API_KEY`), and **codex RELAY** (the Chat-only provider reaches c3's loopback-bound Responses→Chat relay across the container boundary via `host.docker.internal`; the relay base URL's loopback host is rewritten for the container and the per-run relay token is appended into the env-file as `CODEX_API_KEY`). ADR-0024 follow-ups (2026-06-13). |
| SND-R24 | **`networkDisabled` / `readonlyRootfs` are per-workspace security policies, not system-definition fields (ADR-0025).** They live only on the workspace config; the system definition no longer carries them (the parse layer strips any legacy value — first-read discard, no migration). The merge step resolves both from the workspace config, **deny-by-default** (absent ⇒ `networkDisabled` on, `readonlyRootfs` on). The normalisation step persists an explicit off value so a workspace can loosen a policy. `networkAllowlist` stays on the system definition (admin-controlled egress extension point, Phase 2). Note: the `readonlyRootfs` default flipped from off to on — a security-baseline tightening (builds/caches needing a writable root must set the workspace policy to off).      |
| SND-R17 | **Container→host relay hop (codex RELAY, ADR-0024 follow-up):** when `networkDisabled` is off, the container is started with an extra-hosts entry mapping `host.docker.internal` to the host gateway so a sandboxed codex can reach c3's **loopback-bound** relay (no relay-binding widening). Reachability is platform-split: Docker Desktop maps `host.docker.internal` to the host loopback (supported, zero new exposure); Linux-native maps it to the bridge gateway (a known limitation — formal support awaits an in-container relay sidecar). This hop adds no `0.0.0.0` / bridge / LAN bind (constitution C-SEC-5 unchanged by this feature). See codex-relay.md §2.6/§2.7.                                                                                                                      |

## Run Integration (ADR-0024)

The sandbox subsystem is plumbing; the **run lifecycle** decides when to use it. The wiring (2026-06-12):

1. **Gate** — sandbox fires only when a driver + registry are wired AND the run has an isolated effective working directory (a worktree run) AND the workspace's sandbox is enabled with an existing definition. The worktree-only + custom-only invariants are already enforced upstream by the configuration-normalisation step.
2. **Pick** — the agent picker randomly selects one pool agent (a pure step). The resolve callback reports each candidate's id, vendor, and wire API, so a custom codex DIRECT (responses wire) OR RELAY (chat wire) agent is admitted while a system-login codex (no wire API) is not. Rejections — empty pool, unavailable agent, unsupported vendor, unsupported wire — each hard-fail the run.
3. **Pin** — the pick is pinned onto the pending dev session, so every downstream session-launch resolution (the vendor fork, the degradation chain, the SDK launch) resolves to it. The provider connection then reaches the container per vendor: **claude** via the env-file (`ANTHROPIC_*` variables, the unchanged claude wrapper path); **codex DIRECT** via the wrapper's forwarded argv (base URL → the provider base-URL config, model → the model flag) plus `CODEX_API_KEY` mirrored into the env-file; **codex RELAY** via the same wrapper argv (the relay provider config, with the base URL rewritten to `host.docker.internal`) plus the per-run relay token appended into the env-file as `CODEX_API_KEY`, the relay itself staying loopback-bound (SND-R17).
4. **Launch** — the sandbox launch starts the container, mounting the worktree at `/workspace` and the workspace's centralized specs root read/write at its unchanged host absolute path.
5. **Hard-fail** — any failure settles the run as an error and returns; no host bare run (deny-by-default).

## Runtime wiring & host-environment hygiene (2026-06-13 follow-ups)

The ADR-0024 run integration above describes the gate; these are the concrete wiring + host-isolation requirements that make a **real** sandbox run work (each fixed a bug that silently no-op'd or hard-failed the feature):

| ID      | Rule                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| SND-R18 | **Composition-root wiring.** The server builds the registry from the system sandbox definitions (settings, read at startup) and instantiates one Docker runtime, threading both into the run-launch dependencies. Without this the gate's "driver and registry present" precondition is never satisfied and sandbox silently never fires. The Docker client library is bundled as an **external** dependency in the server bundle (it dynamically loads a native addon the bundler cannot inline).                                                                                                                                                |
| SND-R19 | **Daemon socket resolution.** The Docker runtime does NOT rely on the client library's hard-coded `/var/run/docker.sock` (absent on most macOS setups, which select the daemon via the Docker CLI _context_). It resolves in order: the `DOCKER_HOST` environment variable → first existing of `/var/run/docker.sock`, `~/.docker/run/docker.sock` (Docker Desktop), `~/.colima/default/docker.sock` (Colima), `~/.rd/docker.sock` (Rancher) → the client library default. A wrong socket surfaces as a missing-socket error and hard-fails the run.                                                                                              |
| SND-R20 | **Host spawn working dir ≠ container working dir.** The vendor SDK spawns the wrapper **on the host**, so its working directory must be the run's worktree (the effective working directory, a real host dir — and the bind-mount source). The **container** working dir is set independently by the wrapper running the in-container command with the working directory set to `/workspace`; the in-container CLI reports `/workspace` from its own process working dir. Setting the host spawn's working directory to `/workspace` (a container-only path) made the host spawn fail to find the directory ("native binary … failed to launch"). |
| SND-R21 | **Container env hygiene (deny host-loopback).** The container env-file is built from the c3 server's own environment, which leaks host-only values. The wrapper builder **strips proxy variables (`http_proxy` / `https_proxy` / `all_proxy` / …) whose value points at loopback** (`127.0.0.1` / `localhost` / `::1`) — inside the container that loopback is the container itself, so a host-loopback proxy makes every provider call fail with a connection-refused error. Non-loopback proxies are kept. The wrapper's search path must also resolve the `docker` executable (an absolute host location, e.g. `/opt/homebrew/bin`).           |
| SND-R22 | **`IS_SANDBOX=1` for claude.** The claude wrapper environment carries `IS_SANDBOX=1` so the in-container claude accepts the skip-permissions flag despite running as **root** in the image (otherwise it aborts: "cannot be used with root/sudo privileges"). Safe precisely because the run is container-isolated. Unattended worktree dev runs therefore need an auto-approving mode (a bypass-permissions mode): a background run has no live approver, so a gated mode would stall.                                                                                                                                                           |
| SND-R23 | **Worktree under the c3 home, not `$TMPDIR`.** Worktrees live under the c3 home's worktrees area (SND-R14, follow-up) — `$TMPDIR` (`/var/folders` on macOS) is outside Docker Desktop's default file-sharing set, so a worktree there bind-mounts as an **empty** `/workspace`. The c3 home defaults under `$HOME` (always shared) and follows the configured settings location.                                                                                                                                                                                                                                                                  |

- A run reaches that host server over **REST/SSE**, rather than spawning a child the wrapper could swap.

- codex RELAY only crosses the boundary for **LLM provider egress**: the codex CLI **runs inside the container** and does its file/bash work in-container against `/workspace`; the relay routes that one network call back to the host's loopback relay. Isolation holds because the **executor** (the process that touches files and runs commands) is in the container.
- The directory binding breaks too: under the sandbox path the run's working directory is `/workspace` (the **container's** path) — a host server handed a directory of `/workspace` resolves it against the host, where that path does not exist.
