# agent-config — 领域规格

## 概览

agent-config 领域管理会话据以启动的**智能体**。一个智能体是一个厂商无关的
公共外壳(`id`、`vendor`、`configMode`、`displayName`、`enabled?`、`icon?`)加上一个按
`vendor` 判别的 `config` 子对象。有三个厂商拥有真正的适配器 + 配置形态:
(同样的三元组)。Codex 的启动时策略闸门(`sandboxMode`/`approvalPolicy`)——用来
替代其缺失的逐工具审批——**不**持久化在智能体上:它是在启动时根据
会话 `defaultMode` 通过中立映射表推导出来的(AC-R14,2026-06-06-008)。`vendor` 决定
启动哪个客户端(ADR-0011);一个会话可以由这三者中的任意一个**主驱动**(2026-06-06-007)。

与 `vendor` 正交,每个智能体的 **`configMode`** 决定其厂商连接来自哪里:`'system'`
(使用厂商 CLI 自身的配置/登录态——不覆盖 `baseUrl`/`apiKey`;`model`
是一个在两种模式下都会被读取的**独立覆盖项**,2026-07-02-001)或 `'custom'`(应用完整的
厂商三元组)。旧版不可删除、忽略覆盖项的**系统智能体
单例已被淘汰**(2026-06-06-007):`configMode: 'system'` 现在是任意厂商上
按智能体设置的一个选项,每个智能体都可删除 + 可编辑,唯一的不变式是注册表永不为
空,且恰好有一个智能体是**默认**的。配置持久化在 `~/.c3/settings.json`;
按会话的智能体绑定持久化在 `~/.c3/state.json`,作为一个**双键空间**(ADR-0015):一个
可变的 `pendingIntents` map 用于尚未运行的会话,以及 `sessionAgents`**事实**——某个
真实会话实际运行所用的智能体加上其**被冻结的厂商**。一个会话的厂商是一个不可变的
不变式:一旦它产生了一份记录(该记录只存放于该厂商自身的原生存储中,c3
永不存储会话内容),其厂商就不能再改变;在同一厂商内,智能体可以自由
互换。

**范围:** 智能体注册表、默认智能体选择、按会话的智能体绑定,以及
为每次运行解析会话的启动覆盖项。
**边界:** 不驱动 SDK 查询循环(那是 agent-session 的职责),也不持有权限状态。

## 核心实体

| 实体       | 说明                                                                                                                                                                                                                                                 |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --- |
| configMode | `'system'`(使用厂商 CLI 自身的配置——不覆盖 `baseUrl`/`apiKey`;`model` 是一个在两种模式下都会被读取的独立覆盖项)或 `'custom'`(应用 `baseUrl`/`apiKey`/`model`)。与 `vendor` 正交(2026-07-02-001)                                                      |
| 兜底智能体 | 合成的 `{ vendor: 'claude', configMode: 'system' }` 智能体(id 为 `'system'`),仅在首次启动时或 settings 为空/损坏时用作默认种子——**不是**受保护的单例                                                                                                 |
| 系统设置   | 整个配置:智能体 + 默认智能体 id(按项目的旋钮 `defaultMode`/`consensus`/`devSkill`/`maxRoundsPerStage`/`maxSpeechChars` 已移到 `projectConfigs` map 下的 workspace setting 中,见 [workspace-setting](../workspace-setting/workspace-setting-spec.md)) |
| 待定意图   | 一个 `pendingId → { agentId, createdAt }` 条目:尚未运行的会话希望使用哪个智能体。可变;在首次绑定时被复制为一个事实,否则由清理任务回收(AC-R17)                                                                                                        |
| 会话事实   | 一个 `realId → { agentId, vendor }` 条目:某个真实会话实际运行所用的智能体 + 其**被冻结的**厂商;缺失 ⇒ 该会话使用默认智能体(AC-R6)                                                                                                                    |     |

见 [agent-config-models.md](agent-config-models.md)。

## 业务规则

| ID     | 规则                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| AC-R1  | **(已淘汰,2026-06-06-007)** 不再存在受保护/不可删除/忽略覆盖项的系统智能体单例。每个智能体——包括携带旧版 id `'system'` 的那个——都可删除 + 可编辑。`configMode: 'system'`(AC-R13)是逐智能体设定的、以无厂商覆盖项方式启动的方法;id `'system'` 仅作为**合成兜底** id 存续下来(AC-R3)。向后兼容:没有 `configMode` 的旧版持久化智能体在加载时会被视为 `configMode: 'custom'`(AC-R13),因此此前已配置的智能体会以可编辑的厂商字段呈现。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| AC-R2  | `defaultAgentId` 必须指向一个存在且**已启用**的智能体。在保存时(以及每次加载归一化时),一个未知、已被移除,或现在**已禁用**的默认值会被**改写**为按顺序号顺序的**下一个已启用**智能体——共享的解析规则从已禁用默认值的所在位置向前扫描,再折回到整体第一个已启用的智能体;当**没有**任何智能体启用时,回退到合成兜底 id `'system'`(AC-R3)。这是**存储时改写**(持久化的值发生变化,2026-06-15-001),而不仅是运行时解析;web console 在一个智能体被禁用/移除的瞬间就应用同样的规则,使选择器立即反映出来。注册表永不为空(AC-R3),因此默认值总能解析出来。**启动兜底不受影响(AC-R10):** 启动解析按 id 解析一个已绑定会话的智能体,从不按 enabled 过滤,因此绑定到一个已禁用智能体的会话仍能启动——只有**未绑定的**默认值才会被改写。                                                                                                                                                                                                                                                         |
| AC-R3  | `save_settings` 在服务端被归一化:每条记录都由其 `vendor` 分支迁移 + 校验,没有 id 的智能体会得到一个新 uuid,重复的 id 会被丢弃。系统智能体**不会**被重新注入;取而代之的是,如果没有任何有效智能体幸存,就会合成一个单一兜底(`{ id: 'system', vendor: 'claude', configMode: 'system' }`),使注册表永不为空。归一化后的结果会作为 `settings` 回传。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| AC-R6  | 新的(待定)会话默认是未绑定的,因此它们以默认智能体启动(AC-R4)。一个待定会话**可以**携带一个*意图*(AC-R16)——首次运行前选定的期望智能体——此时它会以该智能体启动。启动解析会读取两个键空间,因此一个待定 id 解析为其意图,一个真实 id 解析为其事实。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| AC-R7  | 配置持久化在 `~/.c3/settings.json`;按会话的绑定持久化在 `~/.c3/state.json`。两者都被原子写入,并失败降级为默认值,使 c3 仍能启动。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| AC-R8  | `defaultMode` 是一个按厂商的权限模式 map(按项目,以厂商 id → 模式 token 的 map 形式存储在 workspace setting 中,2026-06-07-017)。每个厂商 key 持有一个在保存时针对该厂商的模式目录校验过的模式 token;一个会话在启动时读取其厂商的条目。一个不在目录中的 token 会降级为该厂商的默认 token 表,因此未知/缺失的 token 永远不会破坏一个会话。旧版单一模式 token 格式(017 之前)在一次性迁移窗口期内被接受,并分发到每个厂商 key。它为一个新会话的模式做种(SR-R6);此后按会话的模式变更(SR-R5)不会改动它。旧版全局系统设置 `defaultMode`(已废弃)仅在一次性迁移窗口期内被接受。                                                                                                                                                                                                                                                                                                                                                                                                         |
| AC-R9  | `maxRoundsPerStage` 是多智能体讨论每阶段的轮次上限(按项目,存储在 workspace setting 中)。保存时会被归一化:一个 ≥ 8 的有限值会被向下取整并保留;一个低于 8 的正值会被夹紧提升到 8;其他情况(缺失、非有限、≤ 0)回退到默认值 12。讨论引擎读取它。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| AC-R10 | 每个智能体都有一个可选的 `enabled` 标志。归一化时它以显式布尔值持久化;**缺省/`true` ⇒ 启用**(向后兼容:没有该字段的旧配置视为启用),只有显式 `false` 才禁用。任何智能体的 enabled 标志都被统一遵守(不再有系统智能体的特殊处理——AC-R1 已淘汰)。每个**智能体列表消费方**只取已启用的智能体:讨论参与者、共识投票者、降级链(链的归一化会剔除已禁用的 id),以及默认智能体选择器(一个已禁用的智能体不能被选为默认)。一个已禁用的智能体会在被禁用的瞬间从这些池中移除,重新启用时恢复。**启动兜底不受影响:** 启动解析从不按 enabled 过滤,因此一个绑定到已禁用智能体——或其默认/系统兜底已被禁用——的会话仍能启动(一个会话永远不会被锁死)。                                                                                                                                                                                                                                                                                                                                               |
| AC-R11 | 每个智能体都有一个可选的 `icon` 字段(一个表情符号或短文本),用于多说话者场景下的展示身份识别。归一化时该值会被**去除首尾空白,去除后为空则 ⇒ `''`,并截断到 16 字符**;任何非字符串值都回退到 `''`(无自定义图标)。**向后兼容:** 没有 `icon` 的旧配置无错误地加载为 `''`。任何智能体的图标都以同样的方式被遵守(不再有系统智能体的特殊处理——AC-R1 已淘汰)。图标**不会**影响任何启动/列表行为:它是一个纯展示字段,因此智能体列表消费方(参与者、投票者、链、选择器)和启动兜底的行为与没有该字段时完全一致。                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| AC-R13 | 每个智能体在公共外壳上携带 **`configMode: 'system' \| 'custom'`**,与 `vendor` 正交(2026-06-06-007)。`'system'` ⇒ 以厂商 CLI 自身的配置启动(不覆盖 `baseUrl`/`apiKey`;`model` 是一个在两种模式下都会被读取的独立覆盖项——2026-07-02-001);`'custom'` ⇒ 应用完整的厂商三元组(AC-R5)。它**只**支配厂商连接。**向后兼容:** 没有 `configMode` 的旧版记录在加载时**默认为 `'custom'`**(因此此前已配置的智能体会以可编辑的厂商字段呈现);`'system'` 只会在控制台中被显式设置。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| AC-R15 | 处于 `configMode: 'custom'` 的 **codex** 智能体通过 c3 的**进程内 Responses→Chat relay**(ADR-0014)访问其厂商,因为 codex 0.137 只讲 Responses API(上游已移除了 chat wire-api 选项),而 DeepSeek/Kimi/MiMo/MiniMax 都只支持 Chat-Completions。该 relay 托管在 c3 自己的 HTTP 服务器上(一个回环端点);每次运行时,驱动器把真实的 base URL + key 注册在一个不透明 token 之后,并针对一个自定义的 model provider 启动 codex(base URL = 该 relay,responses wire-api,禁用 web-sockets 以强制走 HTTP POST + SSE),以该 token 作为 codex 的 API-key 环境变量,并在 no-proxy 列表中增补回环主机。该 relay 把 codex 的 Responses 请求改写为 Chat Completions 请求,并把上游的 Chat SSE 改写回 codex 能解析的 Responses 事件(每个输出作为一个完整的 item,流以一个 completed 事件收尾)。真实的 key 永远不会到达 codex 子进程;该绑定在运行结束时被清除。转换器 + 处理器是**传输层**(ADR-0009 R2);内核只持有一个注入到驱动器中的惰性 relay 句柄。用户的配置不受影响——他们仍然设置真实的厂商 URL。 |
| AC-R16 | 按会话的绑定是一个**双键空间**(ADR-0015)。一个仍处于待定状态的会话持有一个可变的**意图**(`pendingId → { agentId, createdAt }`);一个真实会话持有一个**事实**(`realId → { agentId, vendor }`)。在**首次绑定**时(待定 → 真实会话 id,与运行时重新赋键同时触发),该意图会被复制为一个事实,记录**实际运行所用**的智能体(已应用默认兜底)及其**被冻结的厂商**;随后该意图被丢弃。该冻结操作是**幂等的**——对一个已存在事实的重试/重新绑定永远不会重新冻结厂商。存储层是厂商无关的(厂商作为一个参数跨界传入),因此存储 → agent-config 的边界保持无环(ADR-0009)。                                                                                                                                                                                                                                                                                                                                                                                                                         |
| AC-R17 | 一旦被冻结(AC-R16),一个会话的**厂商是不可变的**——它的记录只存放在该厂商自身的原生存储中,而且**c3 永不存储任何会话内容**。重新定向一个真实会话的智能体**只在同厂商内**才会成功;跨厂商变更会被拒绝(事实不受影响)。本周期**不支持**跨厂商的 Fork / 回放种子(依 ADR-0011 推迟)。**意图消亡永不产生孤儿事实:** 清除一个意图(重新定向为无)与**清理任务**(在启动时以及每小时扫清超过 7 天的待定意图)只触及意图空间。**迁移:** 一个旧版 v1 单一 map 按 key 形态拆分——`pending:` 开头的 key → 意图,其余的 → 冻结为 `vendor: 'claude'` 的事实(多厂商之前唯一的厂商)。                                                                                                                                                                                                                                                                                                                                                                                                                 |

| AC-R18 | **创建时选择智能体 + 控制台呈现。** `create_session` 可以携带一个 `agentId`(web-console WC-R21);当存在时,它会作为新待定会话的**意图**写入(AC-R16),因此首次运行时会以它启动。缺省/空 ⇒ **Auto**——无意图,运行时解析 `defaultAgentId`。`settings` 回复(以及 `session_selected` 中的 vendor 字段)还额外呈现了两个**运行时推导**的值,而持久化的配置本身并不持有:**`hostStatus`**——来自 process-launcher 探测(ADR-0012)的每个厂商宿主 CLI 存在情况(存在标志、二进制名称、已解析路径、安装提示——包括每个已安装二进制的已解析绝对路径,便于诊断面板显示 c3 具体启动的是哪个可执行文件),前端呈现部分依 ADR-0012 D3 推迟;以及**`bindingStats`**——绑定计数(已绑定事实 / 待定意图,ADR-0015),使控制台能够说明默认智能体的变更**不是追溯性的**(已绑定的现有会话保留自己的智能体/厂商;只有未来的会话会采用新的默认值)。 |
| AC-R19 | **同厂商候选规则单一来源**(2026-06-07-001)。“给定厂商下,除被排除的那个之外的每个*已启用*智能体”是三个厂商同质消费方共享的同一条规则:共识投票(投票者只能在同厂商内可比较)、降级链(AS-R22,不同厂商无法 `resume`),以及手动的标题栏智能体切换器(AS-R23)。基于此,**智能体切换解析**返回切换器负载(`session_selected.agentSwitch`):当前智能体 + 其同厂商的同伴(**进一步过滤为宿主二进制存在的厂商**;ADR-0012;由调用方探测宿主是否存在,使 agent-config 保持不依赖启动器),外加一个 `currentUnavailable` 标志。对于一个待定/沟通会话,或者一个当前智能体可用且没有同厂商备选的真实会话,它返回**什么都没有**(无切换器)。 |

| AC-R20 | **用户可控的全局智能体顺序**(2026-06-15-001)。每个智能体携带一个可选的 `order_seq` 数字——这是每个*隐式*“智能体列表”消费方(设置列表、默认/工具智能体下拉框、讨论参与者、共识投票者,以及默认智能体“回退到下一个已启用的”选择器)读取的唯一排序键。归一化时注册表会被**正规化**:(1)合成的/旧版的系统智能体(id 为 `'system'`)被固定在**最前面**,无论其存储值如何;(2)然后是具有显式 `order_seq` 的智能体,按升序;(3)然后任何**缺失**该值的智能体按其当前数组顺序追加在**末尾**。并列情况和缺失分组按数组索引**稳定地**打破,随后 `order_seq` 被重新分配为紧凑的 `0..n`(这同时也去重了重复的位置)。启用中智能体的选择会返回这个顺序(该过滤保留该顺序,并带有防御性的升序排序)。**向后兼容:** 没有 `order_seq` 的旧版配置会按数组(插入)顺序回填,因此现有安装在被重新排序之前保持当前的视觉顺序。**明确排除在外的:** 降级链(AC-R10)是一个独立的**用户编写的有序 id 列表**,其顺序**就是**回退优先级——它在此**不会**按 `order_seq` 重排;启动解析按 id、从不按位置解析启动目标。web console 通过在保存时从数组顺序为 `order_seq` 打上时间戳来持久化一次拖拽重排。 |

| AC-R21 | **后台工具会话的工具智能体**(2026-06-15-001)。`toolAgentId` 指名运行**后台工具会话**——完成度判定和自动化/会话名称推导——的智能体。语义与 `defaultAgentId`(AC-R2)相仿,但有**一个**区别:**空字符串是“跟随默认智能体”的哨兵值**——归一化让它保持为空(从不自动填充为第一个已启用的智能体),运行时通过通用智能体解析器解析它,因此回退链是 `toolAgentId → defaultAgentId → 合成兜底`。一个**非空**的、指向已移除/现已禁用智能体的 `toolAgentId` 会**在存储时被改写**为按 `order_seq` 顺序的下一个已启用智能体,采用与默认值相同的回退机制(AC-R2/AC-R10/AC-R20);当没有任何智能体启用时,解析为兜底 id `'system'`。web console 将其呈现为默认智能体选择器下方的第二个下拉框(按 `order_seq` 排列的已启用智能体,加上一个居首的“跟随默认”选项),并在一个智能体被禁用/移除的瞬间应用同样的回退——但仅当 `toolAgentId` 非空时才这样做。**启动兜底不受影响(AC-R10):** 解析器从不按 enabled 过滤,因此一个工具会话永远不会被锁死。 |

| AC-R22 | **配额限制自动禁用与定时恢复**(2026-06-15-002)。当智能体错误路径发出一个包含可解析 `reset(s) <time>` 值的可恢复配额/会话限制错误(例如 `You've hit your session limit · resets 10:40pm (Asia/Shanghai)`)时,c3 会用系统 `timezone` 解释该重置挂钟时间,立即将该智能体持久化为禁用状态(`enabled: false`),并在重置时刻创建一个内部的一次性自动化以重新启用它。此次写入采用与用户手动切换相同的归一化规则:一个已禁用的 `defaultAgentId` 或非空的 `toolAgentId`/`intentAgentId` 会被改写为按 `order_seq` 顺序的下一个已启用智能体(AC-R2/AC-R21/AC-R23),被恢复的智能体会重新进入所有已启用智能体的候选池。如果重置时间无法解析,c3 不会创建恢复自动化,并保持现有的错误/降级链不变。 |

## 用户场景

- **首次启动:** 给定没有配置文件,当用户打开设置时,应恰好显示一个智能体——
  合成的兜底(`claude` + `configMode: 'system'`)——并且是默认智能体。
- **添加一个智能体:** 给定设置视图,当用户添加一个智能体,选择其**类型**时,
  它会以稳定 id 持久化,并可被选为默认(AC-R13)。Codex 不需要额外的
  sandbox/approval 输入——该闸门由 `defaultMode` 推导得出(AC-R14)。
- **默认智能体驱动新会话:** 给定一个非系统默认智能体,当用户开始
  一个新会话并发送一条提示词时,该智能体的**厂商**客户端会以其
- **默认模式为新会话做种:** 给定一个已配置的 `defaultMode`(例如 `plan`),当
  用户开始一个新会话时,它会以该模式打开(AC-R8,SR-R6);之后切换其模式
  不会改变 `defaultMode`。
- **禁用一个智能体:** 给定若干个智能体,当用户关闭其中一个并保存时,它会
  从讨论参与者池、共识投票者池、降级
  链,以及默认智能体选择器中消失;重新启用它会在各处恢复(AC-R10)。
- **Codex 作为主驱动:** 给定一个被设为默认(或已绑定)的 codex 智能体,当用户发送一条
  提示词时,该会话会分叉到中立驱动器路径,Codex 客户端会以**从会话 `defaultMode`
  推导出的** `sandboxMode`/`approvalPolicy` 作为其启动时闸门运行
  (AC-R14);若 `codex` 宿主 CLI 缺失,该轮会以醒目的“Codex 不可用”
  错误结束,而不是误走 claude 路径启动。
- **已禁用智能体永不锁死会话(反例场景):** 给定一个绑定到某智能体的会话,该智能体
  之后被禁用,当该会话运行时,它必须**永远不会**被阻止启动——
  已绑定/默认兜底仍然适用(AC-R10),且注册表永不为空(AC-R3)。
- **删除所有智能体(注册表永不为空):** 给定用户移除了所有智能体,当设置
  被归一化时,一个单一的兜底智能体(`claude` + `configMode: 'system'`)会被合成
  并成为默认——不存在需要保护的不可删除单例(AC-R1/R3)。
- **悬空的默认值(反例场景):** 移除作为默认值的智能体**绝不能**
  使 `defaultAgentId` 悬空——它会回退到**第一个**剩余的智能体(AC-R2)。
- **旧版配置迁移(向后兼容):** 给定一个在厂商重构之前写入的 `settings.json`
  (扁平的 `name`/`baseUrl`/`apiKey`/`model`,无 `vendor`/`configMode`),当 c3 加载它时,
  每个智能体都会被迁移到 `claude` 分支(`name → displayName`,扁平字段 → `config`),
  `configMode` 默认为 `custom`(因此该智能体已保存的 url/key/model 仍可编辑),并像之前一样启动
  (AC-R12/R13)。
- **未知厂商被丢弃(反例场景):** 给定一个 `vendor` 没有注册 schema 的智能体
  (拼写错误 / 未来厂商),当设置被归一化时,该智能体会被**丢弃**,其
- **首次运行时厂商被冻结:** 给定一个新会话被(通过意图或默认)绑定到一个 codex 智能体,
  当它运行并产生一份记录时,其事实会被冻结为 `vendor: 'codex'`;之后
  尝试将其重新绑定到一个 claude 智能体会被**拒绝**,因此其历史记录仍可从
  codex 存储中读取(AC-R16/R17)。
- **同厂商智能体互换:** 给定一个被冻结为 `claude` 的真实会话,当用户将其重新定向到
  另一个 claude 智能体时,互换会成功,厂商不变(AC-R17)。
- **被遗弃的待定会话被回收(反例场景):** 给定一个已创建但从未发送过提示词的会话,
  当 7 天过去后,清理任务会回收其待定意图,永远不会有孤儿事实
  被创建(AC-R17)。
- **通过拖拽重排智能体:** 给定设置列表中的若干个智能体,当用户把某一行的
  拖拽手柄拖到新的位置并保存时,新的顺序会被持久化(`order_seq` 从
  保存时的数组顺序打上时间戳),系统智能体会被归一化重新固定到最前面,然后该顺序会
  出现在所有展示智能体列表的地方——默认/工具智能体下拉框、讨论
  参与者,以及共识投票者——而用户编写的降级链保留其自己的
  顺序(AC-R20)。向后兼容:一个没有 `order_seq` 的既有配置会保持其当前的视觉
  顺序,直到第一次拖拽为止。
- **选择一个图标:** 给定设置视图,当用户打开某个智能体行的表情符号选择器并
  点击一个表情符号时,它会被写入该智能体的 `icon` 文本字段并像任何手动
  编辑一样保存;手动文本输入框仍然接受自由输入。该选择器是 web console 中一个**纯展示的输入
  辅助控件**——它写回同一个 `icon` 字段,不改变任何
  持久化或归一化逻辑(AC-R11 仍支配去空白/空值/16 字符上限)。

## 领域事件(线上)

消费 `get_settings`、`save_settings`、`load_workspace_setting`、`save_workspace_setting`。发出 `settings`、`workspace_setting`。见
[共享协议](../../../shared/api-conventions/websocket-protocol.md)。

## 交互

- **agent-session** —— 通过启动解析,为每次运行接收已解析的环境覆盖项 / model。
- **session-registry** —— 提供该绑定所依据的活跃会话 id(该
  绑定本身存放在 agent-config 自己的 `~/.c3/state.json` 中,而非注册表的状态)。
- **web-console** —— 渲染完整设置视图页面,并发送 `get_settings` / `save_settings`。
- **discussion** —— 读取 `maxRoundsPerStage` 作为编排循环每阶段的轮次上限
  (AC-R9)。

## 数据字典

- **Agent(智能体)** —— 一个按 `id` 键控的 Claude Code 启动档案。
- **默认智能体** —— 未绑定会话据以启动的智能体。
- **settings.json** —— 位于 `~/.c3/settings.json` 的智能体注册表。
- **state.json(`~/.c3`)** —— 双键的会话→智能体绑定(v2):`pendingIntents`(意图)+
  `sessionAgents`(带冻结厂商的事实);区别于
  `~/.claude/c3/` 下 session-registry 的状态文件。
- **待定意图** —— 一个尚未运行的会话的可变期望智能体条目(`pendingIntents`)。
- **会话事实** —— 一个真实会话的已敲定智能体 + 被冻结的厂商(`sessionAgents`)。
- **厂商归属** —— 一个会话的厂商在首次绑定时被固定下来的不可变不变式。
