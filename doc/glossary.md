# Glossary

Business and technical terms used across c3 doc. Defined once here; domain docs
reference these rather than redefining them.

- **c3**: Code Creative Center. The application: a local web UI that gates Claude Code's tool use through a browser.
- **Agent run**: A single `query()` invocation driven by one user prompt. Streams assistant text, tool activity, and permission requests until it completes or errors.
- **VendorModeCatalog**: The per-vendor declaration of native mode tokens (2026-06-07-012). Each entry pairs a vendor's `token` (a `ModeToken`) with its i18n `labelCode` and the neutral `ActionMode × ToolGate` grid cell it maps to. Each vendor declares its own catalog; the catalogs are aggregated per-vendor as a compile-time-exhaustive set. Shipped to the web on `settings.vendorModes` so the console renders the mode picker per vendor without any vendor-specific branching.
- **Permission decision**: The user's answer to a permission request: `allow` or `deny`.
- **Permission mode**: The policy applied to tool calls in the viewed session. As of 2026-06-07-012 it is a vendor-native mode token carried on the wire as the session's `mode` (on `set_mode` / `mode_changed`), interpreted through the session's vendor VendorModeCatalog as a neutral `ActionMode × ToolGate` grid cell. Claude's former five-value permission-mode union remains as that vendor's token set; the generic type is a plain string mode token. Persists per-session, seeded from the project's `defaultMode`. See also VendorModeCatalog.
- **canUseTool**: The SDK callback c3 supplies. Invoked for sensitive tool calls; c3 routes the question to the browser and returns the resolved decision.
- **Sensitive tool**: A tool the SDK classifies as needing approval under the active mode (e.g. `Write`, `Edit`, dangerous `Bash`). Read-only tools are auto-allowed by the SDK and never produce a request.
- **Auto-deny**: The outcome when a pending permission request is cleared because the run was stopped (`stop_run` / delete / workspace removal) before the user decided. There is no timeout-based auto-deny — an unanswered request blocks indefinitely, and switching the view never denies it.
- **Wire protocol**: The JSON message contract over the `/ws` WebSocket. Two discriminated unions of messages — `ClientToServer` and `ServerToClient` — documented in the [WebSocket protocol contract](shared/api-conventions/websocket-protocol.md).
- **Authentication (认证)**: Establishing **who** a connection is before it may drive agents. The mandatory precondition for binding the server to any non-loopback address (constitution C-SEC-5, ADR-0023). As of 2026-06-11 it is **contract-only**: types, persisted config shape, and wire messages exist; no runtime (middleware/login/hashing/token signing) and no actual relaxation of the localhost-only bind yet. See the [auth domain](domains/core/auth/auth-overview.md).
- **AuthProvider**: The `kind`-discriminated authentication backend abstraction (ADR-0023). `kind: 'basic'` (**BasicAuthProvider** — `accounts: { username, PHC passwordHash }[]` + `adminUsername`: **multiple accounts, exactly one admin**) is runtime-live; every account may sign in, `adminUsername` designates the single config-authority account (no RBAC). `kind: 'oauth'` (**OAuthAuthProvider** — generic OIDC: `issuer`, `clientId`, `clientSecretRef`, `redirectUri`, `scopes`, `usePkce`, `allowedEmails`, `adminEmail`) is **contract-only** (config persists; `/auth/callback`, discovery, PKCE, token exchange, JWKS, session minting all deferred to the OAuth-runtime task, so enabling auth still works only with `basic`); `adminEmail` is the single admin email and must be a member of `allowedEmails`. `basic` and `oauth` are mutually exclusive (one `provider` union). `sso`/multi-tenant remain the reserved extension point (add a `kind` arm + a server zod arm; nothing else changes). The same neutral-abstraction + per-kind-extension shape as the ADR-0011 vendor model.
- **AuthConfig**: The authentication configuration hung on `SystemSettings.auth?` — `{ enabled, provider, session, exposure? }` (ADR-0023). Absent block or `enabled: false` ⇒ no auth (the C-SEC-5 default); a malformed block normalizes (fail-soft) to disabled. `session` is the provider-neutral `AuthSessionPolicy` (token `ttlSeconds` + `signingKeyRef`, a reference to the signing key, never the key itself); `exposure.bindAddress` records network-exposure intent.
- **Session token (会话令牌)**: The provider-neutral issued credential a client presents after login — `AuthSessionToken { tokenId, subject, issuedAt, expiresAt }` (ADR-0023). Minted on successful `login`, carried opaquely on the wire, verified per request (signing/verification deferred to a later task). Distinct from a c3 **Session** (an agent conversation) and from the session-registry's persisted state.
- **passwordHash / signingKeyRef / clientSecretRef**: The "secret-by-indirection" fields of `AuthConfig` (ADR-0023, AUTH-R3/R4/R9). `passwordHash` stores a password only as a PHC hash — never plaintext; the plaintext `AuthLoginRequest.password` exists in transit only. `signingKeyRef` is a _reference_ (env var name / keystore id) to the token signing secret, which is never persisted in the system settings. `clientSecretRef` (on `OAuthAuthProvider`) follows the same discipline: a reference to the OAuth client secret, never the secret value itself; zod strips any stray plaintext `clientSecret` a client round-trips.
- **Workspace directory**: The seed workspace directory passed as `--workspace` (defaults to the current working directory), used as the SDK `cwd`. Claude reads and writes files relative to it. The legacy flag `--project` remains as a deprecated alias for one cycle.
- **Session**: A Claude conversation in a workspace. Its execution is owned by a process-wide **Session Runtime**; a connection only **views** one session at a time. Runs are not bound to a connection (ADR 0006).
- **Works**: The top-level navigation tab that groups a user's development work sessions. Formerly named "Sessions" (renamed 2026-06-08 to disambiguate from the general session concept). Renders as the `console` tab internally. See also **Session**.
- **Work session**: A session displayed under the Works tab — the primary development conversation. Distinguishes a Works-tab session from intent sessions, discussion sessions, and schedule sessions.
- **Session Runtime**: The process-wide owner of one session's run: its abort/handle, a `baseline + buffer` of wire events for replay, the current viewers, and live status. Shared across connections; survives switching/disconnect.
- **SessionKind**: The source-of-truth taxonomy for a run/agent invocation's **business scenario** (which scenario produced it): `work` (general dev session — user console, intent→dev hand-off, automation dev-turn), `intent` (read-only intent-communication session), `discussion` (orchestrator + research), `schedule` (a scheduler-launched, socket-less run), `consensus` (a consensus vote), `tool` (internal tool call — completion judging + title derivation), `spec` (a spec-authoring session, writes confined to the intent's spec directory). Business-source judgements — which scenario may trigger a schedule, which security gate applies — read `sessionKind`. Carried on the session runtime's `sessionKind` field and the `run:started`/`run:settled` events. Split out of the old `RunKind` on 2026-06-26, whose 7 business values moved here verbatim with `'session' → 'work'` (`RunKind` itself was earlier the two-value `'normal' | 'intent'`; `'normal' → 'session'`). **`schedule` is a trigger source, not a run type:** a schedule _triggered_ by a user session reacts to a `work`-kind run; `schedule` only tags the scheduler's own socket-less run. Event-triggered schedules fire only on `sessionKind === 'work'`.
- **RunKind**: The taxonomy for a run's **execution form** (how it executes), orthogonal to **SessionKind** — `interactive` (a socket-backed run a human is watching: user console, intent→dev hand-off, intent/spec comm sessions), `background` (a socket-less run still on the run bus — the automation dev-turn), `headless` (the scheduler's own socket-less run), `internal` (an internal orchestration/tool invocation — discussion, consensus, judge/naming). Carried on the session runtime's `runKind` field and the `run:started`/`run:settled` events. Two runs of the same `sessionKind` can differ here — e.g. a `work` console is `interactive` while a `work` automation dev-turn is `background`. Narrowed out of the old 7-value `RunKind` on 2026-06-26 (its business values moved to **SessionKind**). Currently recorded for audit/extensibility; no consumer branches on it yet.
- **Session status**: A session's live run state: `idle`, `running`, `awaiting_permission`, or `team`. Broadcast to all connections via `session_status` to drive sidebar badges.
- **Viewed session**: The single session a connection currently watches. The next `user_prompt` from that connection runs against it; switching the view never stops the previous session's run.
- **Run handle**: Live controls for an in-flight run handed to the connection handler (a set-permission-mode control), letting a mode change apply mid-run.
- **Interrupt / abort**: Stopping an in-flight run via the SDK's interrupt control. Triggered only by `stop_run`, `delete_session`, or `remove_workspace` — never by switching the viewed session or closing the connection.
- **Static embed**: The web bundle inlined into the compiled binary (generated and gitignored). Served when no filesystem web bundle is present on disk.
- **claude CLI**: The host-installed `claude` executable the SDK spawns to run the agent. Resolved from the `CLAUDE_PATH` env var or the system PATH (via the host-binary launcher). A hard runtime dependency — the Claude instance of the per-vendor **host binary** intent (ADR-0012).
- **turn_end**: The terminal server→client event for one prompt→result turn, with `reason` `complete` (incl. a stopped run) or `error`. Never means the session ended — it stays alive for the next prompt.
- **Intent (意图)**: A project-scoped ledger item — `title`, `content`, `priority` (P0–P3), `status`, optional intra-project dependencies — persisted in the local intent ledger, keyed by the resolved workspace path. The unit the intent list, refine, and launch-development all operate on (ADR 0007). Renamed from **Requirement** (requirements→intents, PR-2). **Disambiguation:** the domain **Intent** (capitalized; a persisted work-unit entity) is NOT the lowercase word _intent_ used elsewhere for unrelated concepts — a Vue child component's event _intent_ (a user-action emit), the ADR-0011 permission _intent_, the consensus vendor-free _intent_, and the orchestrator's next-subtopic / broadcast _intent_. Those are same-word / different-context (cross-referenced here) and are deliberately untouched by this rename.
- **意图级别 (Intent priority)**: An intent's priority, one of `P0`, `P1`, `P2`, `P3` (P0 highest).
- **模块名称 (Module)**: An intent's owning module — a free-text label the communication agent infers from the item's title/content (e.g. 认证、会话、意图管理). Persisted on the intent as a required free-text field defaulting to the empty string; `''` when unidentified or for historical rows. Lays the data groundwork for later module-based organization/display (ADR 0007).
- **New Intent session (Communication session)**: The per-project, long-lived, read-only agent session used to refine ideas into intent items. A real SDK session kept out of the normal session list (the hidden set); runs in forced `default` mode and cannot edit/write/run commands/spawn sub-agents/run slash commands (ADR 0007).
- **save_intents**: The MCP tool (`mcp__c3__save_intents`) the communication agent calls to propose persisting a batch of intents. The save **handler** raises the confirmation itself (emitting the `permission_request` wire frame the gateway transport defines) and writes to the ledger only after the user allows — so a vendor pre-approval that skips `canUseTool` still prompts. Both vendors (claude in-process MCP, codex HTTP MCP) share this one handler-owned gate (ADR 0007). A batch item may carry an optional `intentSessionId` to back-link the intent to the communication session that produced it — but **only when the batch saves exactly one intent** (a multi-item batch ignores it; the store never writes it for `length > 1`). The agent fills it with the session id injected into its prompt; the save handler normalizes that to the bound comm-session id so it resolves via `open_intent_chat`. The schedule-only `save_intent_directly` tool does not expose the field.
- **启动开发 (Launch development)**: The action that turns a `todo` intent into a **background normal session** running the configurable development skill (`devSkill` in system settings; empty by default ⇒ no skill prefix) with the intent content, sets the intent to `in_progress`, and records `lastDevSessionId` for the development back-link. The run never auto-completes the intent.
- **撰写 spec (Write spec)**: The quality-gate action (`write_spec`) that turns an intent into a constrained, reviewable **spec document** under the **fixed, centralized spec root** (`<c3 home>/doc/<project-path-segment>`, isolated per project and shared by all of the project's worktrees), in a dated `yyyy/mm/dd/yyyy-mm-dd-<NNN>-<slug>` directory. The intent's `spec_path` records the **absolute** location (doc live outside the workspace and are not committed to Git). It launches a **write-confined** spec session on the configured spec agent (`specAgentId`) whose writes are limited to the spec directory (the rest of the project is read-only) — it writes the spec, never code. Claude-only (the path-level write lock is a Claude-path mechanism); a non-Claude spec agent is rejected (RM-R21). The spec root is **not** user-configurable.
- **Spec document (规格文档)**: The single source of truth for one change, authored from an intent before development. It describes WHAT and WHY (not implementation code) and is held to a five-dimension self-check: complete, consistent, verifiable, scoped, traceable (RM-R21).
- **自动提交 lint 自愈 (Auto-commit lint self-heal)**: When the automation orchestrator's `done` auto-commit is rejected by a **pre-commit lint hook** (the commit-and-push step reports a commit-hook failure), it self-heals instead of hard-stopping by handing the failure to the **dev agent once** — lint toolchains differ per project, so there is no portable fix command. It resumes the same dev session with the lint error summary, lets the agent fix it, then retries the commit **once**. Any non-lint commit/push failure, or a lint failure surviving the single agent attempt, is a hard stop (RM-A13/RM-A6).
- **Intent ledger**: The local SQLite store for intents in the c3 home directory. Distinct from the session registry's persisted state (under the inherited `.claude` home); accessed through a cross-runtime SQLite driver adapter that fails soft so c3 still boots without it (ADR 0007).
- **Streaming input**: The SDK mode where the prompt is a controlled async-iterable rather than a one-shot string. Pushing a turn appends to the same live session; closing the stream ends the query. c3 runs **all** sessions this way (ADR 0008): it keeps the lead process alive for agent teams, makes mid-run mode changes and interrupts effective, and reproduces one-shot behaviour for normal sessions by closing the stream when the turn produces its result.
- **Agent team**: A Claude Code SDK arrangement where a **team lead** delegates work to **teammates**. c3 detects it at runtime when a team tool (`TeamCreate`, `SendMessage`, or a background `Agent`) is used, upgrading the session to a **persistent session** (AS-R14, ADR 0008). **Claude-locked** (AS-R21, 2026-06-06-006): a session becomes a team only when its vendor has `streamingPush` (a resident lead running in-process `TeamCreate`/`SendMessage`) — only Claude does; a non-Claude session is never marked `team`. A heterogeneous teammate is a deferred capability.
- **Team lead**: The primary agent of an agent team — the `query()` run c3 owns. In a team session its process stays alive between turns (it does not exit on `result`) to coordinate teammates; c3 keeps the run in flight until the user explicitly stops it.
- **Teammate**: A delegated agent spawned by the team lead, e.g. a background `Agent` (`run_in_background: true`) that runs detached and reports its result back to the lead asynchronously. Killing the lead process would orphan teammates — the reason the lead must stay alive (ADR 0008).
- **Persistent session**: A session whose run stays in flight between turns rather than ending each turn. Currently only a `team` session: the lead process is alive between turns, status holds at `team` (not `idle`), and it ends only on explicit user stop (`stop_run` / `delete_session` / `remove_workspace`).
- **team_upgraded**: The one-shot server→client event (and `SessionStatus` `team`) marking a session as a persistent agent team. Recorded in the session buffer so reconnect replay shows it; drives the team badge and the always-enabled composer in the UI.
- **AgentDriver**: The neutral lifecycle + streaming face of a vendor: starting a run yields a handle that exposes its session id, a canonical message stream, and a required abort control, plus capability-gated optional controls. Generalizes the Claude run loop (ADR-0011).
- **AdapterCapabilities**: The probed ledger of a vendor's optional/degradable abilities — two parts (ADR-0011, amended 2026-06-07): **(a)** exactly seven **boolean** live-run flags: `interrupt`, `setActionMode`, `streamingPush`, `inProcessMcp`, `forkSession`, `perToolApproval`, `taskStore`; **(b)** a structured sub-ledger of session-lifecycle capabilities for the operations (`list` / `read` / `resume` / `rename` / `delete`), each a 4-state capability state (`'none'` / `'partial'` / `'full'` / `'temporarily-unavailable'`) — a boolean could not tell `none` (structural NO) apart from `temporarily-unavailable` (mechanism exists, not currently reachable), and that distinction is what the UI renders. Required capabilities carry no flag. The upper layer probes a flag (and method presence) before reaching for a divergent control, degrading when absent; the session sub-ledger's states drive row-action gating by state, with **zero `if (vendor === …)`** (ADR-0011).
- **Canonical message**: The vendor-spanning message envelope (ADR-0011, per the 010 diff): `vendor` tag + `sessionId` (unconditional) + discounted `role`/`blocks`/`ts`/`turnId?` + two-level `vendorExtra` overflow. Blocks are append-with-upsert; a tool's return is **embedded** on its `tool_use` block, not a standalone block. **Promoted to the wire** (SDK-free) in ADR-0013 — the wire gains only a `vendor` dimension, never a per-vendor schema.
- **ActionMode × ToolGate (neutral permission grid)**: The 2-axis replacement for Claude's five-way `PermissionMode` (ADR-0011): `ActionMode{plan,build}` (what the run may do) × `ToolGate{always-ask, on-sensitive, trusted-prefix, never-ask}` (how tools are gated). Each adapter declares a VendorModeCatalog that translates its native mode tokens into this grid — a forward mapping (token → grid) and a reverse, lossy mapping (grid → token). The neutral policy resolves a tool call (name + input + context) to allow / ask / deny. Promoted to the shared wire contract in 2026-06-07-012 as the single SoT.
- **Host-binary launcher**: The vendor-agnostic host-binary probe: it resolves a vendor to an absolute executable path or nothing (precedence: the vendor's `*_PATH` env override → PATH probe → cache), backed by a per-vendor table (binary name / env override / install hint), and probes one or all vendors for the boot health check. The **first capability gate** — an unresolved binary short-circuits before any adapter or capability (ADR-0012).
- **Adapter registry**: Maps each implemented vendor to its factory (only `claude` today) and turns the host-binary probe into the available / missing split. Probes **before** constructing each adapter; an unresolved binary lands in the missing set (with install hint) and is never built. The available set is the kernel's source of truth for available agent types and is logged at boot (ADR-0012).
- **AdapterCapability (wire capability set)**: The wire-facing set of capability names (ADR-0013) naming the seven optional/degradable capabilities: `interrupt`/`setActionMode`/`streamingPush`/`inProcessMcp`/`forkSession`/`perToolApproval`/`taskStore`. The kernel's boolean capability ledger is keyed by exactly these names; a compile-time assertion pins the two so they cannot drift.
- **Block upsert (two-form)**: The rule that lets both vendor message forms share one model (ADR-0013): blocks are keyed by `(sessionId, block.id)` and **upserted**, not append-only. Claude's whole-message frame and Codex's incremental update frames both collapse to "revise in place, don't stack"; a tool's return back-fills its `tool_use` monotonically. Implemented by the canonical accumulator's block-upsert step.
- **c3SessionId**: The opaque, vendor-free session handle (`"c3s_" + sha256(vendor \0 vendorSessionId)[:32]`, ADR-0013) — the only session id that crosses out of the kernel. Deterministic (no persistence needed) and containing neither the vendor name nor the raw id as a substring, so a vendor id never leaks into a URL or storage key. Resolves to a `{ vendor, vendorSessionId }` reference kept inside the kernel.
- **SessionAccessor**: The **read-only** lazy-normalizing union over the available vendors' native session stores (ADR-0013): listing merges across vendors (native id hidden in `vendorExtra`), reading routes to the owning store via a lazily-built c3-handle → reference index. Native stores stay the source of truth — no double-write; the index is a rebuildable runtime cache (存储形态归一、位置不归一).
- **Session→agent binding (two-key)**: The per-session record of which agent a session runs on, split into two key spaces (ADR-0015, agent-config AC-R16): a mutable **pending intent** and a settled **session fact**. Persisted in the registry state (a `pending-intents` space + a `session-agents` space); resolving a session's agent reads both. Distinct from c3SessionId / SessionAccessor, which are about reading transcripts.
- **Pending intent**: The mutable half of the binding (ADR-0015): `pendingId → { agentId, createdAt }` — which agent a not-yet-run session wants. Editable/clearable; copied to a fact at first bind and dropped, else reaped by the janitor after 7 days. Clearing/reaping an intent never touches the fact space, so it can't orphan a fact.
- **Session fact / vendor ownership**: The settled half of the binding (ADR-0015): `realId → { agentId, vendor }` — the agent a real session ran on plus its **frozen vendor**. The vendor is the immutable invariant: a session's transcript lives only in that vendor's native store (c3 never stores session content), so the vendor cannot change. Same-vendor agent swaps are allowed (succeed); cross-vendor changes are rejected. Cross-vendor Fork / **replay-seed** stays deferred (ADR-0011).
- **release (build)**: The thin orchestration layer over the existing build/binary primitives that produces multi-platform artifacts. It does not replace the ordinary web/server build or the single-binary build; it sequences three race-free phases — Phase 0 web build (once) → Phase 1 generate the static embed (once) → Phase 2 parallel binary-compile fan-out (read-only). See the [release spec](non-functional/release.md), ADR-0010.
- **channel**: A named distribution track for release artifacts (e.g. `stable`, `nightly`). Determines artifact naming/suffix and which version is published. **Placeholder** in release 1/7 — the orchestrator builds without a channel concept yet.
- **hardened (tier)**: A build-hardening level passed to the per-target build — the ladder from `default` up through stripped symbols, reproducible builds, signing, and notarization. **Placeholder** in release 1/7: only the `default` tier is honored; other values warn and fall back.
- **version SoT**: The single source of truth for the artifact version and its build-time injection (currently the package manifest; the `--version` flag reports `0.1.0`). A unified version-SoT + git-tag/channel injection is a **placeholder** for a later release wave.
- **license-server (LS)**: The **separate product** that owns the authoritative commercial-entitlement record for c3 — licenses (each carrying its live installation binding), orders, revocations (ADR-0026). Deliberately outside the c3 process (its PostgreSQL / GitHub OAuth / WeChat payment are accepted there, not in c3). c3 is a client of LS. **Not** the c3 [auth](domains/core/auth/auth-overview.md) provider — LS controls _entitlement_, auth controls _access_. See the [product-license domain](domains/commerce/product-license/product-license-overview.md) and the [LS API contract](shared/api-conventions/license-server-api.md).
- **Entitlement**: The c3-side derived answer to "may this installation create new work right now?" — one of `Unactivated` / `Active` / `Grace` / `Expired` / `Disabled` (ADR-0026, product-license PL-R\*). `Active`/`Grace` permit new sessions; the rest **gate** new-session creation. Distinct from **Authentication** (who may connect): entitlement is server-authoritative and independent of the active auth provider.
- **Entitlement token**: The LS-signed, offline-verifiable assertion of entitlement that c3 caches and checks **between heartbeats**. c3 honors `Active` only after verifying its **Ed25519** signature against the **embedded LS public key** (reusing the release-signing anchor, SEC-8), within the token's validity window (product-license PL-R5). Trust comes from the signature, not the network.
- **License key**: The **random, unique** identifier of a license on the c3 ↔ LS API. After GitHub sign-in the user **selects** one of their licenses in the browser to **bind** this installation (product-license PL-R1) — there is no key to copy/paste; the key identifies the chosen license on the wire. A **shareable handle, not a bearer credential** — and binding is **exclusive** (one license, one installation at a time; a rebind displaces the previous one).
- **Alive token**: The **per-binding bearer credential** LS returns **once** when an installation binds a license; c3 presents it on every heartbeat to prove it holds the **live** binding (product-license PL-R2/PL-R3). Rotated on each (re)bind, so a displaced installation's token stops validating (its next heartbeat returns `disabled`); LS stores only its hash. Distinct from the **license key** (a shareable handle) and from a **Session token** (auth) and a Claude credential.
- **Offline grace**: The **30-minute** window after the last successful heartbeat during which c3 keeps treating entitlement as `Active` despite failing heartbeats, bridging transient network/LS outages (product-license PL-R4). After it elapses with no success, entitlement lapses to `Expired` and new-session creation is gated.
- **License badge / license menu**: The c3 web-console surfacing of entitlement state (entitled / grace / expired / unactivated / disabled) and the menu offering activation, status detail, and the purchase link (product-license PL-R7). Informational — it never blocks the UI by itself; enforcement is the gating point.
- **Session gating**: c3's refusal to **create a new session** when entitlement is not `Active`/`Grace` (product-license PL-R6). It is the single enforcement point for licensing. It **never** interrupts an in-flight run or makes an existing session unusable (ADR-0006). Not to be confused with the permission gateway (per-tool gating) or the auth login gate (per-connection access).
- **Service agreement (incl. no-refund terms)**: The 《软件使用授权与服务协议(含无退款条款)》 a user must accept and have recorded **before** proceeding on the LS user flow — at sign-in on the trial path and **on the order** before payment on the renewal path (product-license PL-R9). The no-refund clause is **one term** of this agreement, not the agreement itself: the product is a virtual/digital good that **does not support refunds**, and the MVP ships **no refund workflow** (PL-R10). Disputes/abuse are handled by admin **revocation**, which propagates to c3 via heartbeat (PL-R8).
