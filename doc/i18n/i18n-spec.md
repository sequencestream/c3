# 国际化(i18n)规范

> 适用范围:前端 UI 文案。本规范的命名约定为**冻结点**,是后续
> 全量文案抽取的全局解锁前提 —— 新增 namespace / 改约定须先改本文档再动 key。

## 1. 技术基建

- 库:vue-i18n v11(Composition API,非 legacy 模式)。
- 编译:消息在**编译期预编译**,运行期不携带消息编译器(仅运行时 + 仅 Composition)。
- 基准语种:`en`,作为源语言基线文案文件。回退语种为 `en`。
- 缺 key:缺失/回退时控制台**显式 warning,不静默**。
- 类型安全:消息 schema 取自基准语种结构,由 schema 推导出叶子路径
  的有限并集作为合法 key 类型,并导出受类型约束的翻译函数与组件内翻译访问器,
  **key 拼错在类型检查阶段编译失败**。

> 注:vue-i18n 原生翻译函数 / 模板翻译指令仅提供 key **自动补全**(其签名会把入参推断为
> 字面量,故不会对拼错报错)。需要编译期拦截拼错时,**使用本规范约定的受类型约束的翻译函数 /
> 组件内翻译访问器**,而非直接使用 vue-i18n 原生未约束的翻译函数。

## 2. Key 命名规范(冻结)

key 段结构:

```
<namespace>.<subject>[.<modifier>...][.<suffix>]
```

全小写;段内多词用 `camelCase`,段间用 `.` 分隔。

### 2.1 namespace(第一段,冻结九个)

| namespace    | 用途                                 | 示例                               |
| ------------ | ------------------------------------ | ---------------------------------- |
| `common`     | 跨页复用的通用词(按钮 / 状态 / 动作) | `common.action.save.label`         |
| `nav`        | 导航 / 顶栏 / Tab                    | `nav.refresh.tooltip`              |
| `permission` | 工具调用权限提示                     | `permission.tool.allow.label`      |
| `settings`   | 系统设置                             | `settings.theme.label`             |
| `session`    | 会话                                 | `session.list.refresh.tooltip`     |
| `automation` | 自动化                               | `automation.form.name.placeholder` |
| `discussion` | 讨论                                 | `discussion.input.placeholder`     |
| `intent`     | 需求                                 | `intent.detail.empty`              |
| `error`      | 错误 / 异常文案                      | `error.network.timeout`            |

- 通用词进 `common`,业务词归各自域;错误文案统一进 `error`。
- 需要新 namespace 时,**先在本表登记再加 key**。

### 2.2 后缀约定(可选 —— 表达 UI 角色)

| 后缀           | 含义                | 示例                           |
| -------------- | ------------------- | ------------------------------ |
| `.label`       | 可见标签 / 按钮文字 | `session.title.label`          |
| `.placeholder` | 输入框占位          | `discussion.input.placeholder` |
| `.tooltip`     | 悬浮提示            | `nav.refresh.tooltip`          |
| (无后缀)       | 普通正文 / 消息     | `intent.detail.empty`          |

### 2.3 主语在前(subject-first)

主语(对象 / 实体)在前,动作 / 角色在后:

- ✅ `session.list.refresh.tooltip`、`permission.tool.allow.label`、`intent.detail.empty`
- ❌ `refreshSessionList`、`tooltipForRefresh`、`allowToolPermission`

### 2.4 插值与复数

走 vue-i18n 原生机制:

- 具名插值:消息 `"{count} items"` 配合具名参数 `{ count: 3 }`。
- 复数:消息 `"one item | {count} items"` 配合传入 number 触发分支选择;消息内 `{count}` / `{n}` 自动暴露。

**复数基数随语言变(CLDR cardinal)**,分支数按 locale 不同:

- en:2 分支 `one | other`(默认规则即可)。
- zh / ja / ko:1 分支(无 `|`,单形式;传 number 直接返回该形式并插值)。
- ru:3 分支 `one | few | many`,默认规则错误,须为 ru 注册自定义复数规则(末位 1 非 11→one、末位 2–4 非 12–14→few、其余→many)。

### 2.5 日期 / 数字本地化

绝对日期 / 数字走 vue-i18n 的本地化日期 / 数字格式化(底层为平台 `Intl.DateTimeFormat` / `Intl.NumberFormat`,随 UI 语言):

- 命名预设单一数据源:统一在一处定义日期格式预设(`short`/`full`/`datetime`)与数字格式预设(`integer`/`decimal`),同时供 i18n 配置注册与纯展示工具(纯工具在 Node 单测下直调 `Intl`)。
- 组件内取得本地化日期 / 数字格式化器,按命名预设调用(如 `short` 日期、`integer` 数字);名称冲突时可别名。
- 纯展示工具的格式化与比较函数**显式接收 locale**,保持 Node 可测。

## 3. 用法

- **组件内**:取得受类型约束的组件内翻译访问器,以合法 key 调用翻译函数。合法 key(如
  `common.action.save.label`)正常返回;非法 key(如 `common.save`)在类型检查阶段编译失败。
- **组件外**(composables / 纯工具):import 受类型约束的全局翻译函数(如 `error.network.timeout`)。
- **模板内**:可用 vue-i18n 原生模板翻译指令获得补全(但拼错不报错);需编译期拦截请在脚本里
  用受类型约束的翻译访问器,可见标签与 `title`/`placeholder` 等可译属性均经翻译函数渲染。

## 4. 文件布局

源语言基线文案文件(schema 来源)、i18n 初始化与类型增强(导出受类型约束的翻译函数与组件内
翻译访问器)、应用入口装载 i18n,三者各居一处。

> 空的源语言文案(`{}`)可正常编译与启动;此时合法 key 类型为空集,任何字面量翻译调用
> 都会报错 —— 随基线文案逐步填充自然解锁对应 key。

## 4. 测试 / 抽取桥接约定(冻结)

i18n 抽取会改变所有可见文案;组件测试中**依赖可见英文文案 / `title` / `placeholder` 等
会被翻译的属性的断言必须在抽取前解耦**,否则抽取即全红。两条配套约定如下。

### 4.1 `data-testid`(稳定测试选择器)

- 风格:**组件域-角色 kebab-case** —— `<组件/域>-<角色>`,全小写,`-` 分隔。
  示例:`session-list-refresh`、`session-row-rename`、`discussion-pending`、
  `disc-tab-<kind>`、`settings-save`、`task-more-completed`。
- 仅打在测试需要稳定选择的节点(交互元素、空态/状态文案节点),**不全量铺**。
- 断言**永不**比对可见译文;改为:存在性 / class / 事件 / 数量 /
  fixture 注入的业务数据。
- 组件测试框架无按 testid 取元素的便捷方法;用 `[data-testid="…"]` 属性选择器。

### 4.2 `data-i18n-key`(i18n 待抽取标记)

- 加在"承载 i18n 待抽取文案"的节点(可见文本节点;或文案在 `title` / `placeholder` 上的节点)。
- **本规范下值留空** `data-i18n-key=""` = "此节点文案待抽取";抽取阶段按 §2 命名填入
  真实 key 并接翻译函数(可见文本及 `title` / `placeholder` 等可译属性同理)。
- 抽取待办可由全文检索空值 `data-i18n-key=""` 标记全量枚举;待基线文案填齐后该标记可移除。

## 5. 质量门禁(自动化)

三道闸防硬编码与缺 key / 占位符漂移,串入类型检查 / 提交暂存校验 / CI。

### 5.1 i18n 一致性校验闸

以 `en` 为基准,全量扫描(忽略命令行传入的文件名,故可安全挂提交暂存校验),**四类**校验
(前三类守 web 文案,第四类守服务端 code+params):

| 校验                | 含义                                                                                                                                            | 级别               |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- | ------------------ |
| 覆盖(coverage)      | 各非 `en` locale 必须覆盖 `en` 的全部叶子 key                                                                                                   | 缺 key = **error** |
| 多余 key(extra)     | locale 含 `en` 没有的 key                                                                                                                       | **warn**           |
| 占位符(placeholder) | 非复数 key:两端 `{...}` token 多重集 + 竖线分支数必须一致。复数 key(en 含 `\|`):分支数随 locale(CLDR),仅比对去除 `{count}`/`{n}` 后的占位符集合 | 篡改 = **error**   |
| code→key            | 前端代码中字面量引用的 key 必须存在于基线文案                                                                                                   | 缺失 = **error**   |
| 动态 key            | 变量 key / 含插值的模板串,无法静态判定                                                                                                          | **warn**(跳过)     |
| 未使用 key          | 基线文案中从未被字面量引用的 key                                                                                                                | **warn**           |
| code→locale(SoT)    | 服务端 code 单一数据源中每个 code 的 `key` 须在基线文案;声明的 `params` 须与该 key 占位符一致                                                   | 不符 = **error**   |
| code 越界           | 服务端发送的展示 code 须登记于单一数据源                                                                                                        | 未登记 = **error** |
| 未发送 code         | 单一数据源中从未被服务端发送的 code                                                                                                             | **warn**           |

退出码:有 **error → 非零(CI 红)**;仅 warn → 零(绿)。**空基线文案且无翻译调用 → 绿**。

占位符校验采用「变量→不可译 token→比对」:把每条消息里的 `{name}`/`{0}`/ICU 块
(`{count, plural, …}`,嵌套花括号整块保留)抽成 token 多重集,翻译端重命名 / 增删 token
即报错——保护 `{name}` `{n}` 不被改坏。

**复数感知**:当 base(en)为复数(含 `|`)时,复数基数是语言相关的(en=2、ru=3、zh/ja/ko=1),
故**不比对分支数**,改为比对去除 `{count}`/`{n}` 后的「去重占位符集合」(`{count}` 可能只出现在部分
分支,如 en `one item` 无 `{count}`)。base 非复数时仍按严格多重集 + 分支数=1,堵截误加竖线。

### 5.2 模板硬编码文案禁令(no-raw-text)

对所有 web 单文件组件模板启用硬编码文案禁令,禁模板中
中英文硬编码文案(纯标点 / 数字 / 符号经豁免规则放行)。

> **当前级别 = `error`(阻断)**。M1 全量抽取已完成(全部组件可见静态文案 100% 走翻译函数,
> web 端 0 raw-text),前置条件满足,已于 2026-06-04 由 `warn` 切 `error` —— 此后任何
> 新增硬编码文案即 lint / CI 红。
>
> 注:数据绑定的枚举/动态显示(如 `{{ status }}`)与绝对日期/数字(平台本地化字符串等)不被
> 该禁令捕获,属另行跟踪的本地化专项,不在本闸范围。

### 5.3 接入点

- 类型检查:受约束的翻译函数在类型检查阶段拦截拼错 key(见 §1)。
- 提交暂存校验:提交触及 web 前端源文件时跑 i18n 一致性校验闸。
- CI:类型检查 → lint → i18n 校验,任一非零即红
  (lint 步不把 warning 视为失败,故硬编码禁令的 warn 现阶段不卡 CI)。

## 6. 文案抽取工具

半自动抽取全量可见文案、产出源语言基线的工具。**抽取是半自动的**:工具只给
「候选 key → 原文」草稿,**key 命名与插值切分由人工/团队定稿**;基线文案定稿后 freeze 再分发。

### 6.1 两种模式

| 模式       | 行为                                                                                                |
| ---------- | --------------------------------------------------------------------------------------------------- |
| 扫描(默认) | 全量解析 web 单文件组件的**模板文本节点** + **可译属性**,产出**稳定**候选清单(生成物)               |
| 合并       | 读人工定稿的「点分 key → 原文」映射,展开为嵌套对象并**合并**进基线文案(冲突即报错,**绝不静默覆盖**) |

扫描产出**确定性**(按 文件 → 行 → 列 排序 + 同文件同文案去重计 occurrences):同一棵未变更的代码树两次运行**字节一致**。命令行忽略多余入参(可安全挂提交暂存校验),始终全量扫。候选清单是生成物、不入库,随时重新扫描即可重生。

### 6.2 可译属性白名单(穷举兜漏)

可译属性 = `title`/`placeholder`/`alt`/`label`/`badge`/`content` +
`aria-label`/`aria-description`/`aria-placeholder`/`aria-roledescription`/`aria-valuetext`。
**刻意排除** ARIA 状态/关系属性(`aria-hidden`/`aria-expanded`/`aria-pressed`/`aria-haspopup`/
`aria-modal` 等)—— 它们载布尔/ID/token 而非文案,不可笼统前缀匹配 `aria-*`。

### 6.3 候选项类别(kind)与人工处理

| 类别          | 含义                               | 人工处理                                                                                                           |
| ------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| 文本          | 模板文本节点                       | 直接定稿 key                                                                                                       |
| 静态属性      | 静态可译属性(`title="…"`)          | 直接定稿 key                                                                                                       |
| 字面量绑定    | 绑定到字符串字面量(`:title="'…'"`) | 直接定稿 key                                                                                                       |
| 字面量插值    | `{{ '…' }}` 字面量插值             | 直接定稿 key                                                                                                       |
| 动态(dynamic) | 变量 / 三元 / 拼接 / 模板串        | **人工切分**:三元的每个字面量分支拆成独立 key;纯变量绑定不产 key;含变量的模板串切成具名插值(如 `"Open chat: {x}"`) |

草稿 key 形如 `<namespace>.<subject>.<slug>[.<suffix>]`(见 §2),namespace 按文件路径启发式落到冻结九段;命不中 → `common` 且标记为待人工复核归属。slug、suffix 同为草稿。

### 6.4 freeze → 分发流程

1. 扫描产候选清单 → 人工据 §2 定稿 key、切分动态项、复核归属猜测、剔除代码/环境标识符等非译项。
2. 合并进基线文案(或手工编辑)→ 团队定稿措辞 → **freeze 基线文案**(源语言基线)。
3. 以基线文案为 schema,新增非 en 语种文件并分发翻译;一致性校验闸守覆盖率与占位符不漂移。

> 当前基线:基线文案已由本工具产出首版骨架(243 key,源语言覆盖率 100%,一致性校验绿)。
> 组件模板尚未接翻译函数(留作下游接线任务),故现阶段全部 key 在一致性校验中表现为「未使用」warning(不卡 CI)。

#### 6.4.1 freeze 机制

freeze 由工具落地,基线文案的 SHA-256 记入冻结清单(随仓库提交),配套三项能力:

- **freeze** — 写/刷新冻结清单(刻意编辑基线文案后须重跑)
- **unfreeze** — 删冻结清单(回到"未冻结"WARN 态)
- **校验冻结** — 独立校验;并已内联进一致性校验闸头部:hash 漂移即 ERROR(CI 红),无冻结清单仅 WARN

刻意改基线文案的正确姿势:unfreeze → 改 → freeze,把新 hash 一并提交。

### 6.5 M2 第二批语种(ja / ko)上线 + 逐语言 DoD 开关

第二批语种的"译文已落地"与"是否出现在 UI 下拉"是**两件事**,由两道闸分开管控:

1. **译文落地**:从基线文案拷骨架后逐值翻译,接进 i18n 的消息集。一致性校验守零缺 key + 占位符守恒。术语/禁译走 [`i18n-terms.md`](./i18n-terms.md)(已含 ja/ko 列)。
2. **进 UI 下拉**:由「已启用语言」集合控制;系统设置的语言下拉按是否启用过滤渲染。未进集合的语种,即便译文已加载也不暴露给用户。

**人校信号 = 人工已复核标记**:在 locale 文案顶部加双下划线包裹的顶层元数据键 `__humanReviewed__: true`(扁平化时跳过它,不算 key,不进 vue-i18n)。约定:**模型只读不写此标记**,由人在母语校对通过后手工编辑翻转;翻转后该语种才进「已启用语言」集合。

> **「已启用语言」派生(已接线)**:该集合在**构建期**从各 locale 的**原始导入对象**(剥除元数据之前,故人工已复核标记仍在)派生 —— en/zh 为**无条件基线**(已审的 M1 默认,不做 flag 门控,避免缺标志锁死基准语),其余受支持语种当且仅当原始对象顶层人工已复核标记为真才进集合。除 en/zh 基线外无硬编码语言名;另有专门步骤负责把双下划线元数据从 vue-i18n 所见消息里删掉。派生逻辑由单测覆盖(标记 true→启用、缺失/false→不启用、en/zh 恒启用)。

> 当前状态(2026-06-04):ja/ko 机翻 + 占位符校验已过、译文已接线,四语一致性校验 0 错;CJK 渲染经 headless 截图验证零 tofu、长 hint 无溢出。**但 ja/ko 尚未经母语校对**,故未进「已启用语言」集合,下拉仍只放 en/zh。等人校翻人工已复核标记后再下放。

## 7. 服务端 code+params 协议(单一数据源)

> 适用范围:**服务端 → 前端展示**的错误/通知/toast。目标:消除服务端与 web 双份译文漂移
> —— 翻译只存 web 语言包,服务端永不传译文。服务端日志/调试、发给 LLM 的 prompt 文案
> **保留英文,不纳入 i18n**。

### 7.1 原则

- 服务端对前端展示项**只回传机器可读 `{ code, params }`**(承载于协议错误字段,见
  [`../shared/api-conventions/websocket-protocol.md`](../shared/api-conventions/websocket-protocol.md)),绝不回传译文。`params` 值可含英文技术细节(异常串)—— 属调试数据。
- web 收 `code` → 经**单一数据源**映射到 `error.*` key → 以该 key 与 `params` 调翻译函数。
- 协议层 key 与数据结构保持**英文常量不译**;`code`、`key` 皆英文。

### 7.2 单一数据源(SoT)

一处登记 code→key(及允许的插值名)的展示码映射表;合法 code 类型由该表的键推导,服务端
发送展示项因此类型安全。web 直接复用同一映射(共享代码,无运行期独立映射文件)。**这是唯一权威**;
基线文案与服务端发送点都对它对齐。

### 7.3 构建期生成 + 校验

- 从单一数据源确定性派生一份排序后的 code→key 映射(生成物、不入库,同候选清单模式)
  —— 供查阅/文档,**非第二数据源**。
- 一致性校验闸第四类校验(§5.1 末三行)守 单一数据源 ↔ 基线文案 ↔ 服务端 三方一致,已接 CI:
  发未登记 code、key 缺失、params 与占位符漂移 → **CI 红**。

### 7.4 落地节奏

- Loop 1(基建先行):协议错误字段、单一数据源、生成能力、第四类校验、2 样板 code。
- Loop 2(全量迁移,**已完成**):服务端全部可见 error 点(24 code)迁到 code+params,
  协议错误字段**已移除自由文本 message、转必填结构化 code**。服务端全部错误发送点均为结构化 code。
- 仍属后续(spec 003 §9):web 端非组件代码硬编码、硬编码禁令扩至非组件代码、枚举/原因码
  display label。
