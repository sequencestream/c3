# Non-Functional — Availability

c3 is a local foreground process. There is no clustering, no failover, no SLA. Availability
requirements are about clean degradation and not leaving the user stuck.

## Requirements

| ID      | Requirement                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| AVAIL-1 | If the `claude` executable cannot be found or the SDK errors, the turn ends with a `turn_end` of `reason: 'error'` carrying the message — never a silent hang.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| AVAIL-2 | A permission request blocks indefinitely until the user answers — by design, mirroring the CLI prompt (no timeout). It is released only by a user decision or by the run being stopped (`stop_run` / delete / workspace removal; see AVAIL-4), which resolves it as `deny`. Switching the viewed session or closing the socket never releases it.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| AVAIL-3 | Closing the WebSocket only unsubscribes that connection's view; in-flight runs continue in the background and the server stays up for new connections. Reconnecting and selecting the session replays the full record (ADR 0006).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| AVAIL-4 | Stopping a run (`stop_run` / delete / workspace removal) interrupts it cleanly; `interrupt()` rejections (e.g. "not ready for writing") are swallowed and do not crash the process. A second prompt for a session whose turn is in flight is rejected, not coalesced.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| AVAIL-5 | An unparseable client message is ignored without tearing down the connection.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| AVAIL-6 | The client keeps the socket alive with a 25s `ping`/`pong` heartbeat and detects a half-open link via a 10s pong timeout. On any drop it auto-reconnects with exponential backoff (1s → cap 30s, jittered), then re-selects the active session so history + buffered live events replay without a manual page reload (builds on AVAIL-3).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| AVAIL-7 | A normal session whose **agent run** drops the transport (`socket connection was closed unexpectedly`) auto-`resume`s the **same** run at most **once** per turn, after a bounded 3–5s backoff (status `reconnecting`), preserving full context (AS-R18). This is **bounded by construction** — no unbounded retry / billing loop. Auto-resume is gated by the tool side-effect inference (AS-R19): an unclosed write-class `tool_use` refuses auto and falls back to a manual continue. When auto-resume is refused, disabled (`socketAutoResume: false`, default true), or the single retry is exhausted, the turn **must** emit a terminal `turn_end { reason: 'error' }` and settle to `idle` (run finalization is the backstop — AVAIL-1) — never a silent hang. The auto-resume reuses the same `runId` (single live instance), so it cannot race the liveness reconciler. Distinct from AVAIL-6, which reconnects the **browser↔server WebSocket**; AVAIL-7 reconnects the **server↔SDK** agent run. |

## Observability — file logging

For after-the-fact troubleshooting of a background process, c3 persists its own runtime output to
disk, not just the terminal.

| ID    | Requirement                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ----- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| OBS-1 | The main process tees its console output to a live log file `c3.log` under the c3 home log directory (`~/.c3/log/`, following the same `--settings` / `C3_DIR` override as the rest of c3 home). Every real log **line** is prefixed with a host-local second-precision timestamp `YYYY-MM-DD HH:mm:ss ` (same local-calendar reading as archival, no timezone dependency); the terminal and the file receive the identical prefixed text. Prefixing is per line, not per write call: a line split across writes gets one prefix, and each newline within a write starts a new one. Non-goal: no millisecond precision, and call sites that already print their own time are not de-duplicated. |
| OBS-2 | Logs are archived by **host local calendar day** (deliberately not the configured scheduler timezone, so logging keeps working even when settings fail to load). At a day boundary the live `c3.log` is renamed to `c3-YYYY-MM-DD.log` for the day that just ended, and a fresh `c3.log` continues. A startup catch-up archives a leftover live log from an earlier day (covers a process that spanned midnight while stopped).                                                                                                                                                                                                                                                                 |
| OBS-3 | Archives older than **30 days** are pruned; an archive dated exactly 30 days ago is kept (boundary inclusive). The live `c3.log` and any non-archive file are never pruned.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| OBS-4 | Logging is **best-effort**: a missing/unwritable log directory, a failed archive rename, or a failed prune is caught and reported to stderr, never thrown — the main process must not crash on a logging fault, and must not fail silently.                                                                                                                                                                                                                                                                                                                                                                                                                                                     |

**Non-goals (documented limits):** no size-based rotation (the live file is a single fixed name written
continuously); the 30-day retention and the daily cadence are not user-configurable; no remote log
shipping; scope is the c3 **main process'** own output only — sandbox containers, child processes, and
vendor CLI output are not captured.

## Known gaps (documented, not yet addressed)

- **No runtime eviction.** Session runtimes (their event buffers) live for the process
  lifetime; there is no memory cap or eviction yet. Acceptable for a local single-user tool.
