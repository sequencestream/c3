# 非功能需求 — 可用性

c3 是一个前台进程。没有集群、没有故障转移、没有 SLA。可用性需求关注的是优雅降级,而不是让用户卡住无法动弹。

## 需求

| ID      | 需求                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| AVAIL-1 | 若 `claude` 可执行文件找不到,或 SDK 报错,该轮次以携带错误信息的 `turn_end`(`reason: 'error'`)结束——绝不静默挂起。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| AVAIL-2 | 一个权限请求会无限期阻塞,直到用户作出决定——这是设计使然,与 CLI 的提示行为一致(无超时)。它只能通过用户决策或运行被停止(`stop_run` / 删除 / 工作区移除;见 AVAIL-4)来释放,后者会被解析为 `deny`。切换所查看的会话或关闭 socket 都不会释放它。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| AVAIL-3 | 关闭 WebSocket 只会取消该连接的视图订阅;进行中的运行会继续在后台执行,服务端也会为新连接保持在线。重新连接并选择该会话会重放完整记录(ADR 0006)。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| AVAIL-4 | 停止一次运行(`stop_run` / 删除 / 工作区移除)会将其干净地中断;`interrupt()` 的拒绝(例如“not ready for writing”)会被吞掉,不会使进程崩溃。对一个正在进行轮次的会话发出第二次 prompt 会被拒绝,而不是合并。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| AVAIL-5 | 无法解析的客户端消息会被忽略,不会导致连接被拆除。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| AVAIL-6 | 客户端通过 25 秒的 `ping`/`pong` 心跳来保持 socket 存活,并通过 10 秒的 pong 超时来检测半开连接。一旦连接断开,便以指数退避(1 秒 → 上限 30 秒,带抖动)自动重连,然后重新选择当前活动会话,使历史记录和缓冲的实时事件得以重放,无需手动刷新页面(建立在 AVAIL-3 之上)。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| AVAIL-7 | 一个普通会话的 **智能体运行** 若因传输层断开(`socket connection was closed unexpectedly`)而中断,系统会在有界的 3–5 秒退避后(状态为 `reconnecting`)对 **同一个** 运行进行至多 **一次** 自动 `resume`,并保留完整上下文(AS-R18)。这是**构造上有界**的——不存在无界重试/计费循环。自动恢复受工具副作用推断(AS-R19)的门控:一个未闭合的写类 `tool_use` 会拒绝自动恢复,退回为手动 continue。当自动恢复被拒绝、被禁用(`socketAutoResume: false`,默认 true),或唯一一次重试已耗尽时,该轮次**必须**发出终态的 `turn_end { reason: 'error' }` 并归于 `idle`(运行终结机制作为兜底——AVAIL-1)——绝不静默挂起。自动恢复复用同一个 `runId`(单一活动实例),因此不会与存活性协调器产生竞争。与 AVAIL-6 的区别在于:AVAIL-6 重连的是 **浏览器↔服务端** 的 WebSocket;AVAIL-7 重连的是 **服务端↔SDK** 的智能体运行。 |

## 可观测性 — 文件日志

为便于对后台进程进行事后排障,c3 会将自身的运行时输出持久化到磁盘,而不仅仅是终端。

| ID    | 需求                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ----- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| OBS-1 | 主进程会将其控制台输出同步写入(tee)c3 主目录日志目录下(`~/.c3/log/`,遵循与 c3 主目录其余部分相同的 `--settings` / `C3_DIR` 覆盖规则)的一个实时日志文件 `c3.log`。每一条真实的日志**行**都带有本机本地、精确到秒的时间戳前缀 `YYYY-MM-DD HH:mm:ss `(与归档采用相同的本地日历读法,不依赖时区);终端与文件收到完全相同的带前缀文本。前缀是按行添加的,而非按写调用添加:跨多次写入拼成的一行只会得到一个前缀,一次写入中的每个换行符都会开启新的一行。非目标:不提供毫秒精度,已经自行打印时间的调用点不会被去重。 |
| OBS-2 | 日志按**本机本地日历日**归档(刻意不采用配置的调度器时区,以便在设置加载失败时日志仍能继续工作)。在一天的边界处,实时的 `c3.log` 会被重命名为刚结束那一天的 `c3-YYYY-MM-DD.log`,并接续一个全新的 `c3.log`。启动时的追赶逻辑会归档来自更早一天、遗留下来的实时日志(覆盖跨越了午夜、期间进程处于停止状态的情形)。                                                                                                                                                                                              |
| OBS-3 | 超过 **30 天** 的归档会被清理;恰好 30 天前的归档会被保留(边界含边界当天)。实时的 `c3.log` 以及任何非归档文件永远不会被清理。                                                                                                                                                                                                                                                                                                                                                                              |
| OBS-4 | 日志记录是**尽力而为**的:日志目录缺失/不可写、归档重命名失败、或清理失败都会被捕获并报告到 stderr,而不是抛出——主进程绝不能因日志故障而崩溃,也绝不能静默失败。                                                                                                                                                                                                                                                                                                                                             |

**非目标(已记录的限制):** 没有基于大小的轮转(实时文件是持续写入的单一固定名称);30 天保留期与每日节奏都不可由用户配置;没有远程日志投送;范围仅限于 c3 **主进程** 自身的输出——沙箱容器、子进程以及厂商 CLI 的输出均不在采集范围内。

## 已知缺口(已记录,尚未解决)

- **没有运行时驱逐机制。** 会话运行时(其事件缓冲区)会存活整个进程生命周期;目前尚无内存上限或驱逐机制。在当前的使用规模下可以接受。
