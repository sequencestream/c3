# 0020 — 沙箱驱动作为独立 kernel 模块

- **Status:** accepted
- **Date:** 2026-06-09
- **Driver:** 沙箱隔离需求 → 容器生命周期管理 → 独立于 agent 抽象

## Context

c3 需要在容器中运行 agent 的 vendor CLI 进程，以实现文件系统、网络和资源的隔离。这引入了一个新的基础设施关注点：**容器生命周期管理**——创建、启动、停止、删除容器，以及在容器内执行命令。

在沙箱功能设计初期，有三种可能的归属方案：

1. 将容器操作内嵌到 agent 适配层（每个 vendor 的适配器自行管理容器）。
2. 放在装配胶水层作为粘合代码。
3. 创建一个独立的 kernel 模块。

三个候选各有trade-off。此外，未来可能需要支持多种容器后端（Docker、gVisor、Kata、Firecracker），接口需要足够通用以容纳不同的运行时。

已有架构约束（ADR-0009）要求 kernel 模块不得反向依赖 features/transport，且 kernel 内部模块之间也应有清晰的单向边界。

## Options Considered

### 1. 嵌入 agent 适配层

_Pro:_ 适配器可以精确控制容器的生命周期，与 vendor 进程的 fork 时机完美对齐。
_Con:_ 每个适配器都要实现（或复制）相同的容器管理逻辑——违反 DRY。
_Con:_ 新的 vendor 接入时必须理解容器管理逻辑，增大了接入成本。
_Con:_ 容器后端的选择不应该与 vendor 绑定（一个项目应该能用同一个沙箱配置运行任意 vendor）。
_Con:_ 违反单一职责——适配器的职责是桥接 vendor API，不是管理容器。

### 2. 装配胶水层粘合代码

在装配胶水层中将容器管理作为 run 启动的前置/后置步骤。

_Pro:_ 不需要新模块，改动最小。
_Con:_ 容器生命周期逻辑游离在分层架构之外，没有明确的归属。
_Con:_ 难以测试——装配层是 composition root，不应包含业务逻辑。
_Con:_ 新的容器后端支持需要修改 composition root，违反开闭原则。

### 3. 独立 kernel 模块 (selected)

创建专用的沙箱 kernel 模块，对外暴露统一的沙箱驱动接口。

_Pro:_ 职责清晰——容器生命周期属于 kernel 基础设施层。
_Pro:_ 可替换后端——接口抽象化后，Docker/gVisor/Kata/Firecracker 可互换。
_Pro:_ 遵循 ADR-0009——kernel 模块之间只能单向依赖，沙箱模块不依赖 agent 或 transport。
_Pro:_ 可测试——接口 mock 化后，依赖沙箱模块的上层可以独立测试。
_Con:_ 增加了模块数量（新增模块 + 测试）。

### 4. 事件驱动（通过事件总线管理容器）

将容器的启动/停止作为事件总线上的事件，由独立的订阅者处理。

_Pro:_ 完全解耦——run 生命周期发布事件，沙箱订阅者响应。
_Con:_ 容器的生命周期需要与 run 生命周期严格同步（先启动容器再 fork vendor 进程），事件模型的异步性引入竞态。
_Con:_ 过度设计——对于启动 + 停止两个同步操作，事件总线增加了不必要的复杂度。
_Con:_ ADR-0022 已确定容器生命周期通过运行时句柄上的清理闭包同步管理，无需事件。

## Decision

**采纳选项 3**：沙箱驱动作为一个独立的 kernel 模块存在，通过明确的接口与 run 生命周期集成。

### 接口契约

沙箱驱动提供 7 项能力：

| 能力     | 用途                       | 同步/异步        |
| -------- | -------------------------- | ---------------- |
| 启动     | 创建并启动容器             | async → 容器句柄 |
| 停止     | 停止并可选删除容器         | async → 无返回   |
| 执行     | 在容器内执行命令并收集输出 | async → 执行结果 |
| 流式执行 | 在容器内执行命令并流式输出 | async → 输出流   |
| 快照     | 将容器提交为新镜像         | async → 镜像 ID  |
| 文件拷出 | 从容器复制文件到宿主机     | async → 无返回   |
| 健康检查 | 检查容器健康状态           | async → 健康状态 |

### 与 Run 生命周期的集成

由 ADR-0022 确定，通过运行时句柄上的三个字段承载：

- 运行中的容器句柄
- wrapper 脚本的临时目录
- 容器清理闭包

集成行为：

- 在 fork vendor 进程前启动沙箱，将结果写入运行时句柄
- run 收尾或运行时移除时调用清理闭包，回收容器
- 容器启动失败是**非致命的**——降级为告警日志，run 继续在宿主机执行

### 可插拔后端

沙箱驱动接口对外暴露上述 7 项能力（启动、停止、执行、流式执行、快照、文件拷出、健康检查），各能力均以容器句柄为操作对象。任何容器后端只要实现这一组能力即可接入。

Phase 1 只有 Docker 后端实现。Phase 2 计划加入 gVisor、Kata、Firecracker 实现。

## Consequences

### 正面

- **职责清晰**: 容器生命周期管理有明确归属，与 agent 抽象层完全解耦。
- **可替换后端**: 新增容器运行时只需实现沙箱驱动接口，无需修改上层代码。
- **遵循 ADR-0009**: 沙箱 kernel 模块不依赖 features 或 transport，不引入 WebSocket / HTTP 语义。
- **可测试**: Docker 后端接受可注入的容器客户端实例，沙箱注册表是纯内存操作，沙箱启动器接受 mock 驱动。所有核心逻辑都有单元测试覆盖。
- **优雅降级**: 容器启动失败不阻断 run 执行——run 启动路径捕获错误并降级到宿主机路径。

### 负面

- 新增模块的初始化成本：composition root 需要创建 Docker 后端与沙箱注册表，并注入应用上下文。
- Phase 1 仅支持 Docker 后端。gVisor/Kata/Firecracker 需要额外的适配工作和安全配置。

### 迁移

- 现有的 run-lifecycle 路径已经集成了沙箱路径（git HEAD 的状态）。
- 新增后端时不需要修改现有的 run-lifecycle 路径，只需在沙箱模块下新增对应后端的实现。

## Compliance

- 沙箱 kernel 模块不得导入 features 或 transport 模块（ADR-0009 R1，由导入边界 lint 规则强制）。
- 类型检查必须在所有沙箱类型改动后保持绿色。
- 测试必须覆盖沙箱注册表、沙箱启动器、Docker 后端三处的单元测试。
- 沙箱配置中的类型一致性断言必须保持绿色（验证 schema 与 TypeScript 接口同步）。
