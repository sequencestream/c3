# sandbox ŌĆö Domain Spec

## Overview

The sandbox domain provides **container-based isolation** for agent execution. Rather than running vendor CLI processes directly on the host, the sandbox layer creates lightweight containers with configurable resource limits, filesystem isolation, network controls, and seccomp security profiles.

This is a **kernel infrastructure** domain (inner domain, ADR-0009) ŌĆö it provides the plumbing that higher-level features (runs, schedules, permissions) consume through the {@link SandboxDriver} interface. It knows nothing about agents, sessions, or business logic.

**Scope:** sandbox type definitions, driver interface, configuration schema, named definition registry, Docker runtime implementation, seccomp profile loading/merging.
**Boundary:** it does not manage containers directly (the driver does), does not decide which sandbox to use (the caller does), and does not implement container orchestration (Kubernetes, Swarm).

## Run Integration (ADR-0024)

The sandbox subsystem is plumbing; the **run lifecycle** (`kernel/run/run-lifecycle.ts`) decides when to use it. The wiring (2026-06-12):

1. **Gate** ŌĆö sandbox fires only when a driver + registry are wired AND `rt.effectiveCwd` is set (a worktree run) AND `getProjectSandbox(workspacePath)` is enabled with an existing def. The `worktree-only` + `custom-only` invariants are already enforced upstream by `normalizeSandboxConfig` (config layer).
2. **Pick** ŌĆö `pickSandboxAgent(agentIds, resolve)` (pure, `kernel/run/sandbox-agent.ts`) randomly selects one pool agent. The `resolve` callback reports `{ id, vendor, wireApi }` (`wireApi` via `launchForAgent`), so a custom codex DIRECT (`responses`) OR RELAY (`chat`) agent is admitted while system-login codex (`wireApi` absent) is not. Rejections (`empty-pool` / `unavailable` / `unsupported-vendor` / `unsupported-wire`) each hard-fail the run.
3. **Pin** ŌĆö `setSessionAgent(runId, pickedId)` pins the pick onto the pending dev session, so every downstream `resolveSessionLaunch(runId)` (the vendor fork, the degradation chain, the SDK launch) resolves to it. The provider connection then reaches the container per vendor: **claude** via the env-file (`ANTHROPIC_*`, the unchanged claude wrapper path); **codex DIRECT** via the wrapper's `"$@"` argv (`baseUrl`ŌåÆ`--config openai_base_url`, `model`ŌåÆ`--model`) plus `CODEX_API_KEY` mirrored into the env-file (`codexDirectSandboxEnv`); **codex RELAY** via the same wrapper argv (the `-c model_providers.c3relay.*` config, with `base_url` rewritten to `host.docker.internal`) plus the per-run relay token appended into the env-file as `CODEX_API_KEY` (`codexRelaySandboxEnv`), the relay itself staying loopback-bound (SND-R17).
4. **Launch** ŌĆö `launchSandbox(driver, registry, workspacePath, effectiveCwd)` starts the container, mounting the worktree at `/workspace`.
5. **Hard-fail** ŌĆö any failure settles the run as an error and returns; no host bare run (deny-by-default).

> Supported in the container today: **claude**, **codex DIRECT** (`wireApi=responses`), and **codex RELAY** (`wireApi=chat`, via `host.docker.internal` on Docker Desktop; Linux-native is a known limitation awaiting an in-container relay sidecar ŌĆö SND-R17, codex-relay.md ┬¦2.6). ADR-0024 follow-ups (2026-06-13). Still unsupported (each hard-fails on pick): **system-login codex** (no injected creds in the container) and **opencode**.

## Runtime wiring & host-environment hygiene (2026-06-13 follow-ups)

The ADR-0024 run integration above describes the gate; these are the concrete wiring + host-isolation requirements that make a **real** sandbox run work (each fixed a bug that silently no-op'd or hard-failed the feature):

| ID      | Rule                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| SND-R18 | **Composition-root wiring.** `server.ts` builds the `SandboxRegistry` from `getSystemSandboxes()` (settings, read at startup) and instantiates one `DockerDriver`, threading both into `LaunchRunDeps`. Without this the gate's `deps.sandboxDriver && deps.sandboxRegistry` is never satisfied and sandbox silently never fires. `dockerode` is marked **external** in the server bundle (it dlopens a native `.node` via ssh2ŌåÆcpu-features that esbuild cannot bundle).                                                                                                                        |
| SND-R19 | **Daemon socket resolution.** `DockerDriver` does NOT rely on dockerode's hard-coded `/var/run/docker.sock` (absent on most macOS setups, which select the daemon via the `docker` CLI _context_). It resolves in order: `DOCKER_HOST` env ŌåÆ first existing of `/var/run/docker.sock`, `~/.docker/run/docker.sock` (Docker Desktop), `~/.colima/default/docker.sock` (Colima), `~/.rd/docker.sock` (Rancher) ŌåÆ dockerode default. A wrong socket surfaces as `ENOENT /var/run/docker.sock` and hard-fails the run.                                                                               |
| SND-R20 | **Host spawn cwd ŌēĀ container cwd.** The vendor SDK spawns the wrapper **on the host**, so its `cwd` must be the run's worktree (`effectiveCwd`, a real host dir ŌĆö and the bind-mount source). The **container** working dir is set independently by the wrapper's `docker exec -w /workspace`; the in-container CLI reports `/workspace` from its own process cwd. Setting the SDK `cwd` to `/workspace` (a container-only path) made the host spawn fail with `ENOENT` ("native binary ŌĆ” failed to launch").                                                                                    |
| SND-R21 | **Container env hygiene (deny host-loopback).** The container env-file is built from the c3 server's `process.env` (via `buildChildEnv`), which leaks host-only values. `createSandboxWrapper` **strips proxy vars (`http(s)_proxy`/`all_proxy`/ŌĆ”) whose value points at loopback** (`127.0.0.1`/`localhost`/`::1`) ŌĆö inside the container that loopback is the container itself, so a host-loopback proxy makes every provider call fail with `ConnectionRefused`. Non-loopback proxies are kept. The wrapper PATH must also resolve `docker` (absolute on the host, e.g. `/opt/homebrew/bin`). |
| SND-R22 | **`IS_SANDBOX=1` for claude.** The claude wrapper env carries `IS_SANDBOX=1` so the in-container `claude` accepts `--dangerously-skip-permissions` despite running as **root** in the image (otherwise it aborts: "cannot be used with root/sudo privileges"). Safe precisely because the run is container-isolated. Unattended worktree dev runs therefore need an auto-approving mode (e.g. `bypassPermissions`): a background run has no live approver, so a gated mode would stall.                                                                                                          |
| SND-R23 | **Worktree under the c3 home, not `$TMPDIR`.** Worktrees live at `<c3-home>/worktrees/...` (SND-R14, follow-up) ŌĆö `$TMPDIR` (`/var/folders` on macOS) is outside Docker Desktop's default file-sharing set, so a worktree there bind-mounts as an **empty** `/workspace`. The c3 home defaults under `$HOME` (always shared) and follows `--settings`/`C3_DIR`.                                                                                                                                                                                                                                  |

## opencode ŌĆö Explicitly Unsupported (decision record, ADR-0024 follow-up 2026-06-13)

opencode is **not container-supported, by decision** ŌĆö this is a recorded non-goal, not a missing wire-up. A random pick of an opencode agent hard-fails the run via `pickSandboxAgent` ŌåÆ `unsupported-vendor` (SND-R16).

**Why the wrapper isolation model does not apply.** claude / codex containerization rests on a **per-run child process + wrapper binary-swap**: the sandbox wrapper is `exec docker exec --env-file <f> -i -w /workspace <cid> <vendor-cli> "$@"`, so each run spawns a vendor CLI child, the wrapper swaps that child for in-container execution, and provider credentials ride a per-run env-file. opencode's process model is fundamentally different:

- opencode is a **host-resident server** (`opencode serve`) whose lifecycle is owned by `OpencodeSupervisor` (`kernel/agent/adapters/opencode/supervisor.ts`) ŌĆö in managed mode c3 spawns and health-checks/self-heals it, and **runs reuse the same server process**.
- Its **provider config is server-level, injected once at boot** via `OPENCODE_CONFIG_CONTENT` ŌĆö not per-run.
- A run reaches that host server over **REST/SSE**, rather than spawning a child the wrapper could swap.

So "the binary a run uses" _is_ that host server ŌĆö there is **no per-run child process for the wrapper to replace**, and the wrapper model (swap-child + per-run env-file) simply does not fit opencode.

**Why the codex-RELAY hop cannot be reused (container opencode ŌåÆ host server).** A tempting idea is "let the in-container opencode reach the host-resident server via `host.docker.internal`, like codex RELAY." The direction is backwards and it **defeats sandbox isolation**:

- codex RELAY only crosses the boundary for **LLM provider egress**: the codex CLI **runs inside the container** and does its file/bash work in-container against `/workspace`; the relay routes that one network call back to the host's loopback relay. Isolation holds because the **executor** (the process that touches files and runs commands) is in the container.
- opencode splits **client** from **executor**: tool calls (read/write/edit/**bash**) actually execute in the **host `opencode serve` process**; c3 is only a REST/SSE client + approval gateway (`task-store.ts` notes c3 is OBSERVE-ONLY). There is **no opencode process in the container**.
- So even with an in-container opencode client pointed at the host server, the file/bash side effects still land in the **host server process** on the **host filesystem** ŌĆö the container becomes a hollow shell and sandbox containment drops to zero. **RELAY isolates the network; opencode needs to isolate the executor, and the executor is on the host.**
- The directory binding breaks too: under the sandbox path `driverCwd = rt.sandboxHandle ? '/workspace' : ŌĆ”` (`run-via-driver.ts`), and `/workspace` is the **container's** path ŌĆö a host server handed `directory: '/workspace'` resolves it against the host, where that path does not exist.

Bottom line: RELAY is reusable only when "the executor is already in the container and only the network is missing." opencode is missing the **executor's location**, not the network ŌĆö so the direction must invert: put the **server in the container** (hostŌåÆcontainer), i.e. the sketch below, not a containerŌåÆhost relay.

**Future support sketch (not implemented ŌĆö anchor for a separate intent).** Sandboxing opencode requires replacing the whole launch path: ŌæĀ run `opencode serve` **inside the container** (an in-container resident server, not on the host); ŌæĪ change the host-side REST/SSE client to **reach the container's mapped port** (container networking / port forwarding ŌĆö analogous to codex RELAY's `host.docker.internal` hop but in the hostŌåÆcontainer direction); Ōæó make `OpencodeSupervisor` **container-aware** (in-container spawn / health-check / process-tree kill, with provider config via the container's native env/config instead of host `OPENCODE_CONFIG_CONTENT`). This spans supervisor + launch + config-injection ŌĆö too large for this phase.

**Decision:** this phase **accepts that opencode agents are unavailable under sandbox**; a random opencode pick stays a hard-fail (`unsupported-vendor`, message points to ADR-0024). No in-container `opencode serve` is implemented here.
