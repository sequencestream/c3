# agent-config — Domain Spec

## Overview

The agent-config domain manages the **agents** sessions launch with. An agent is a vendor-agnostic
public shell (`id`, `vendor`, `configMode`, `displayName`, `enabled?`, `icon?`) plus a
`vendor`-discriminated `config` sub-object. Three vendors have real adapters + config shapes:
(same triple). Codex's launch-time policy gate (`sandboxMode`/`approvalPolicy`) — the substitute
for its missing per-tool approval — is **not persisted on the agent**: it is derived at launch from
the session `defaultMode` via the neutral grid (AC-R14, 2026-06-06-008). The `vendor` selects which
client launches (ADR-0011); a session can be **primary-driven** by any of the three (2026-06-06-007).

Orthogonal to `vendor`, each agent's **`configMode`** decides where its provider connection comes
from: `'system'` (use the vendor CLI's own config/login — no `baseUrl`/`apiKey`/`model` override) or
`'custom'` (apply that provider triple). The old undeletable, override-ignoring **system agent
singleton is retired** (2026-06-06-007): `configMode: 'system'` is now a per-agent option on any
vendor, every agent is deletable + editable, and the only invariant is that the registry is never
empty and exactly one agent is the **default**. The configuration persists at `~/.c3/settings.json`;
the per-session agent binding persists at `~/.c3/state.json` as a **two-key space** (ADR-0015): a
mutable `pendingIntents` map for sessions that have not run yet, and `sessionAgents` **facts** — the
agent a real session actually ran on plus its **frozen vendor**. A session's vendor is an immutable
invariant: once it has produced a transcript (which lives only in that vendor's native store — c3
never stores session content), the vendor cannot change; within the same vendor the agent can be
swapped freely.

**Scope:** the agent registry, the default-agent selection, the per-session agent binding, and
resolving a session's launch overrides for each run.
**Boundary:** it does not drive `query()` (agent-session) and does not own permission state.

## Core entities

| Entity         | Description                                                                                                                                                                                                                                                             |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --- |
| configMode     | `'system'` (use the vendor CLI's own config — no provider override) or `'custom'` (apply `baseUrl`/`apiKey`/`model`). Orthogonal to `vendor`                                                                                                                            |
| Fallback agent | The synthesized `{ vendor: 'claude', configMode: 'system' }` agent (`id === 'system'`) used only as the default seed on first boot or when settings are empty/corrupt — NOT a protected singleton                                                                       |
| SystemSettings | The whole configuration: `agents[]` + `defaultAgentId` (per-project knobs `defaultMode`/`consensus`/`devSkill`/`maxRoundsPerStage`/`maxSpeechChars` moved to `WorkspaceSetting` under `projectConfigs` map, see [system-config overview](../system-config-overview.md)) |
| Pending intent | A `pendingId → { agentId, createdAt }` entry: which agent a not-yet-run session wants. Mutable; copied to a fact at first bind, else reaped by the janitor (AC-R17)                                                                                                     |
| Session fact   | A `realId → { agentId, vendor }` entry: the agent a real session ran on + its **frozen** vendor; absent ⇒ the session uses the default agent (AC-R6)                                                                                                                    |     |

See [models.md](models.md).

## Business rules

| ID     | Rule                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| AC-R1  | **(retired, 2026-06-06-007)** No protected/undeletable/override-ignoring system-agent singleton exists. Every agent — incl. one carrying the legacy `id === 'system'` — is deletable + editable. `configMode: 'system'` (AC-R13) is the per-agent way to launch with no provider overrides; `SYSTEM_AGENT_ID` survives only as the **synthesized fallback** id (AC-R3). Back-compat: a legacy persisted agent without `configMode` loads as `configMode: 'custom'` (AC-R13), so previously-configured agents surface with editable provider fields.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| AC-R2  | `defaultAgentId` must reference an existing **enabled** agent. On save (and every load `normalize`), an unknown, removed, or now-**disabled** default is **rewritten** to the **next enabled** agent in `order_seq` order — the shared `resolveDefaultAgentId(agents, current)` scans forward from the disabled default's position, then wraps to the first enabled agent overall; when **no** agent is enabled it falls back to `SYSTEM_AGENT_ID` (the synthesized fallback id, AC-R3). This is **rewrite-on-store** (the persisted value changes, 2026-06-15-001), not a runtime-only resolution; the web console applies the same rule the instant an agent is disabled/removed so the picker reflects it immediately. The registry is never empty (AC-R3), so a default always resolves. **Launch fallback is unaffected (AC-R10):** `resolveSessionLaunch`/`resolveAgent` resolve a bound session's agent by id and never filter on `enabled`, so a session bound to a disabled agent still launches — only the _unbound_ default is rewritten.                                                                                                                                                                                                                                              |
| AC-R3  | `save_settings` is normalized server-side: each record is migrated + validated by its `vendor` arm, agents without an id get a fresh uuid, duplicate ids are dropped. The system agent is **not** re-injected; instead, if no valid agent survives, a single fallback (`{ id: 'system', vendor: 'claude', configMode: 'system' }`) is synthesized so the registry is never empty. The normalized result is echoed as `settings`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| AC-R6  | New (pending) sessions are unbound by default, so they launch with the default agent (AC-R4). A pending session **may** carry an _intent_ (AC-R16) — a desired agent chosen before its first run — in which case it launches with that agent instead. `resolveSessionLaunch` reads both key spaces, so a pending id resolves to its intent and a real id to its fact.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| AC-R7  | The configuration persists at `~/.c3/settings.json`; the per-session binding at `~/.c3/state.json`. Both are written atomically and fail soft to defaults so c3 still boots.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| AC-R8  | `defaultMode` is a per-vendor permission-mode map (per-project, stored as `Record<VendorId, ModeToken>` in `WorkspaceSetting`, 2026-06-07-017). Each vendor key holds a `ModeToken` validated at save time against that vendor's `VendorModeCatalog`; a session at launch reads its vendor's entry via `getDefaultMode(workspacePath, vendor)`. A token absent from the catalog degrades to the vendor's `defaultToken` grid, so an unknown/absent token never breaks a session. The legacy single `ModeToken` format (pre-017) is accepted during the one-shot migration window and distributed to each vendor key. It seeds a new session's mode (SR-R6); per-session mode changes thereafter (SR-R5) do not alter it. The legacy global `SystemSettings.defaultMode` (deprecated) is accepted only during the one-shot migration window.                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| AC-R9  | `maxRoundsPerStage` is the per-stage round cap for multi-agent discussions (per-project, stored in `WorkspaceSetting`). On save it is normalized: a finite value ≥ 8 is floored and kept; a positive value below 8 is clamped up to 8; anything else (missing, non-finite, ≤ 0) falls back to the default 12. The discussion engine reads it via `getMaxRoundsPerStage()`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| AC-R10 | Each agent has an optional `enabled` flag. On normalize it persists as an explicit boolean; **absent/`true` ⇒ enabled** (back-compat: old configs without the field count as enabled), only an explicit `false` disables. Any agent's `enabled` is honoured uniformly (no special system-agent casing — AC-R1 retired). Every **list-of-agents consumer** takes only enabled agents: discussion participants, consensus voters, the degradation chain (`normalizeDegradationChain` drops disabled ids), and the default-agent picker (a disabled agent cannot be picked as default). A disabled agent is removed from these pools the moment it is disabled and restored when re-enabled. **Launch fallback is unaffected:** `resolveSessionLaunch`/`resolveAgent` never filter on `enabled`, so a session bound to a disabled agent — or whose default/system fallback is disabled — still launches (a session is never locked out).                                                                                                                                                                                                                                                                                                                                                             |
| AC-R11 | Each agent has an optional `icon` field (an emoji or short text) used for display identity in multi-speaker contexts. On normalize the value is **trimmed, empty-after-trim ⇒ `''`, and capped to 16 chars**; anything non-string falls back to `''` (no custom icon). **Back-compat:** old configs without `icon` load as `''` without error. Any agent's `icon` is honoured the same way (no special system-agent casing — AC-R1 retired). `icon` does NOT affect any launch/list behavior: it is a display-only field, so list-of-agents consumers (participants, voters, chain, picker) and launch fallback behave exactly as without it.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| AC-R13 | Each agent carries **`configMode: 'system' \| 'custom'`** on the public shell, orthogonal to `vendor` (2026-06-06-007). `'system'` ⇒ launch with the vendor CLI's own config (no `baseUrl`/`apiKey`/`model` override, no claude third-party workaround); `'custom'` ⇒ apply the provider triple (AC-R5). It governs ONLY the provider connection. **Back-compat:** a legacy record without `configMode` **defaults to `'custom'`** on load (so previously-configured agents surface with editable provider fields); `'system'` is only ever set explicitly in the console.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| AC-R15 | A **codex** agent in `configMode: 'custom'` reaches its provider through c3's **in-process Responses→Chat relay** (ADR-0014), because codex 0.137 speaks only the Responses API (the `wire_api = "chat"` option was removed upstream) while DeepSeek/Kimi/MiMo/MiniMax are Chat-Completions-only. The relay is hosted on c3's own Hono server (a loopback endpoint); per run the driver registers the real `{baseUrl, apiKey}` behind an opaque token and launches codex against a custom `model_provider` (`base_url` = the relay, `wire_api = "responses"`, `supports_websockets = false` to force HTTP POST + SSE), with the token as `CODEX_API_KEY` and `NO_PROXY` augmented with loopback hosts. The relay rewrites the codex Responses request into a Chat Completions request and the upstream Chat SSE back into the Responses events codex parses (each output as a full item in `response.output_item.done`, the stream closed with `response.completed`). The real key never reaches the codex subprocess; the binding is evicted on run end. The translator + handler are **transport** (`transport/codex-relay/`, ADR-0009 R2); the kernel holds only an inert relay handle injected into the driver. The user's configuration is unchanged — they still set the real provider URL. |
| AC-R16 | The per-session binding is a **two-key space** (ADR-0015). A still-pending session holds a mutable **intent** (`pendingId → { agentId, createdAt }`); a real session holds a **fact** (`realId → { agentId, vendor }`). At the **first bind** (pending → real `sessionId`, fired alongside the runtime re-key) the intent is copied into a fact recording the agent that **actually ran** (default-fallback already applied) and its **vendor is frozen** (`freezeSessionAgent` → `bindSessionAgent`); the intent is then dropped. The freeze is **idempotent** — a retry/re-bind onto an existing fact never re-freezes the vendor. The storage layer (`kernel/config`) is vendor-blind (vendor crosses in as an argument), so the `config → agent-config` boundary stays acyclic (ADR-0009).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| AC-R17 | A session's **vendor is immutable** once frozen (AC-R16) — its transcript lives only in that vendor's native store, and **c3 never stores any session content**. Re-targeting a real session's agent (`setSessionAgent` → `changeSessionAgentFact`) succeeds **only within the same vendor**; a cross-vendor change is rejected (`{ ok: false }`, fact untouched). Cross-vendor Fork / replay-seed is **not supported this cycle** (deferred per ADR-0011). **Intent death never orphans a fact:** clearing an intent (re-target to none) and the **janitor** (`cleanupStalePendingIntents`, sweeping `pendingIntents` older than 7 days at boot + hourly) touch only the intent space. **Migration:** a legacy v1 single map is split by key shape — `pending:` keys → intents, all others → facts frozen to `vendor: 'claude'` (the only pre-multi-vendor vendor).                                                                                                                                                                                                                                                                                                                                                                                                                              |

| AC-R18 | **Choosing an agent at creation + console surfacing.** `create_session` may carry an `agentId` (web-console WC-R21); when present it is written as the new pending session's **intent** (AC-R16) via `setPendingIntent`, so the first run launches with it. Absent/empty ⇒ **Auto** — no intent, the run resolves `defaultAgentId`. The `settings` reply (and `session_selected.vendor`) additionally surface two **runtime-derived** values the persisted config does not hold: **`hostStatus`** — each vendor's host-CLI presence from the ProcessLauncher probe (ADR-0012; `present`/`binary`/`path`/`installHint`, including the resolved absolute path of each installed binary so the diagnostics panel can show which executable c3 launches), the front-end surfacing ADR-0012 D3 deferred; and **`bindingStats`** — the binding counts (`bound` facts / `pending` intents, ADR-0015), so the console can state that a default-agent change is **not retroactive** (existing bound sessions keep their own agent/vendor; only future sessions adopt the new default). |
| AC-R19 | **Same-vendor candidate rule is single-sourced** (2026-06-07-001). `sameVendorEnabledAgents(vendorScope, excludeId)` — every _enabled_ agent of `vendorScope` except `excludeId` — is the one rule the three vendor-homogeneous consumers share: consensus voting (`vendorScopedVoters`, voters are comparable only within a vendor), the degradation chain (AS-R22, a different vendor can't `resume`), and the manual title-bar agent switcher (AS-R23). Built on it, **`resolveSessionAgentSwitch(sessionId, presentVendors)`** returns the switcher payload (`session_selected.agentSwitch`): the current agent + its same-vendor peers **further filtered to host-binary-present vendors** (ADR-0012; the caller probes via `probeAll`, keeping `agent-config` free of the launcher), plus a `currentUnavailable` flag. It returns **null** (no switcher) for a pending/comm session, or a real session whose current agent is available and has no same-vendor alternative. |

| AC-R20 | **User-controlled global agent order** (2026-06-15-001). Each agent carries an optional `order_seq: number` — the single sort key every _implicit_ "list of agents" consumer reads (the settings list, the default/tool-agent dropdowns, discussion participants, consensus voters, and the default-agent "fall through to the next enabled one" picker). On normalize the registry is **regularized** by `canonicalizeAgentOrder`: (1) the synthesized/legacy system agent (`SYSTEM_AGENT_ID`) is pinned to the **front** regardless of its stored value; (2) then agents with an explicit `order_seq` ascending; (3) then any **missing** one appended at the **tail** in its current array order. Ties and the missing group break **stably** by array index, and `order_seq` is then reassigned to a dense `0..n` (which also dedupes duplicate positions). `enabledAgents()` returns this order (filter preserves it, with a defensive ascending sort). **Back-compat:** a legacy config without `order_seq` is backfilled by array (insertion) order, so existing installs keep their present visual order until re-ranked. **Explicitly excluded:** the `degradationChain` (AC-R10) is an independent **user-authored ordered id list** whose sequence IS the fallback priority — it is **not** re-sorted by `order_seq`; and `resolveSessionLaunch`/`resolveAgent` resolve a launch target by id, never by position. The web console persists a drag-reorder by stamping `order_seq` from array order at Save time. |

| AC-R21 | **Tool agent for background tool sessions** (2026-06-15-001). `toolAgentId` names the agent that runs **background tool sessions** — the completion judge (`judgeCompletion`) and the schedule/session-name derivation (`generateScheduleName`). Semantics mirror `defaultAgentId` (AC-R2) with ONE difference: an **empty string is the "follow the default agent" sentinel** — `normalize` leaves it empty (never auto-fills it to the first enabled agent), and the runtime resolves it through `resolveAgent` so the fall-through is `toolAgentId → defaultAgentId → SYSTEM_AGENT_ID → synthesized` (`resolveToolAgent`/`resolveToolSessionLaunch`). A **non-empty** `toolAgentId` pointing at a removed/now-disabled agent is **rewritten on store** to the next enabled agent in `order_seq` order via the same `resolveDefaultAgentId` fall-through the default uses (AC-R2/AC-R10/AC-R20); when no agent is enabled it resolves to `SYSTEM_AGENT_ID`. The web console offers it as a second dropdown below the default-agent picker (enabled agents by `order_seq` + a leading "follow default" option) and applies the same fall-through the instant an agent is disabled/removed — but only when `toolAgentId` is non-empty. **Launch fallback is unaffected (AC-R10):** the resolver never filters on `enabled`, so a tool session is never locked out. |

| AC-R22 | **Quota-limit auto-disable and scheduled recovery** (2026-06-15-002). When the agent error path emits a recoverable quota/session-limit error that includes a parseable `reset(s) <time>` value (for example `You've hit your session limit · resets 10:40pm (Asia/Shanghai)`), c3 interprets the reset wall-clock in `SystemSettings.timezone`, immediately persists that agent as `enabled: false`, and creates an internal one-shot schedule at the reset instant to set `enabled: true` again. The same normalize rules apply to this write as to a user toggle: a disabled `defaultAgentId` or non-empty `toolAgentId` is rewritten to the next enabled agent in `order_seq` order (AC-R2/AC-R21), and the restored agent re-enters all enabled-agent candidate pools. If the reset time cannot be parsed, c3 does not create a recovery schedule and leaves the existing error/degradation chain unchanged. |

## User scenarios

- **First boot:** Given no config file, When the user opens settings, Then exactly one agent —
  the synthesized fallback (`claude` + `configMode: 'system'`) — is shown and is the default.
- **Add an agent:** Given the settings view, When the user adds an agent, picks its **type**
  is persisted with a stable id and may be chosen as default (AC-R13). Codex needs no extra
  sandbox/approval input — that gate is derived from `defaultMode` (AC-R14).
- **Default agent drives new sessions:** Given a non-system default agent, When the user starts
  a new session and sends a prompt, Then that agent's **vendor** client launches with its
- **Default mode seeds new sessions:** Given a configured `defaultMode` (e.g. `plan`), When the
  user starts a new session, Then it opens in that mode (AC-R8, SR-R6); switching its mode later
  leaves `defaultMode` unchanged.
- **Disable an agent:** Given several agents, When the user toggles one off and saves, Then it
  disappears from the discussion participant pool, the consensus voter pool, the degradation
  chain, and the default-agent picker; re-enabling it restores it everywhere (AC-R10).
- **Codex as primary driver:** Given a codex agent set as default (or bound), When the user sends a
  prompt, Then the session forks to the neutral driver path and the Codex client runs with the
  `sandboxMode`/`approvalPolicy` **derived from the session `defaultMode`** as its launch-time gate
  (AC-R14); if the `codex` host CLI is missing the turn ends with a loud "Codex is unavailable"
  error rather than mis-launching on the claude path.
- **Disabled agent never locks a session (anti-scenario):** Given a session bound to an agent that
  is later disabled, When the session runs, Then it must **never** be blocked from launching — the
  bound/default fallback still applies (AC-R10), and the registry is never empty (AC-R3).
- **Delete every agent (registry never empty):** Given the user removes all agents, When settings
  are normalized, Then a single fallback agent (`claude` + `configMode: 'system'`) is synthesized
  and becomes the default — there is no undeletable singleton to protect (AC-R1/R3).
- **Dangling default (anti-scenario):** Removing the agent that was the default must **never**
  leave `defaultAgentId` dangling — it falls back to the **first** remaining agent (AC-R2).
- **Legacy config migrates (back-compat):** Given a `settings.json` written before the vendor
  refactor (flat `name`/`baseUrl`/`apiKey`/`model`, no `vendor`/`configMode`), When c3 loads it,
  Then each agent is migrated to the `claude` arm (`name → displayName`, flat fields → `config`)
  with `configMode` defaulting to `custom` (so the agent's saved url/key/model stay editable), and launches as before
  (AC-R12/R13).
- **Unknown vendor dropped (anti-scenario):** Given an agent whose `vendor` has no registered schema
  (a typo / future vendor), When settings are normalized, Then that agent is **dropped** and its
- **Vendor frozen at first run:** Given a new session bound (by intent or default) to a codex agent,
  When it runs and produces a transcript, Then its fact is frozen to `vendor: 'codex'`; a later
  attempt to re-bind it to a claude agent is **rejected**, so its history stays readable from the
  codex store (AC-R16/R17).
- **Same-vendor agent swap:** Given a real session frozen to `claude`, When the user re-targets it to
  another claude agent, Then the swap succeeds and the vendor is unchanged (AC-R17).
- **Abandoned pending reaped (anti-scenario):** Given a session that was created but never sent a
  prompt, When 7 days pass, Then the janitor reaps its pending intent and no orphan fact is ever
  created (AC-R17).
- **Reorder agents by drag:** Given several agents in the settings list, When the user drags a row's
  grip handle to a new slot and saves, Then the new order is persisted (`order_seq` stamped from
  array order at Save), the system agent is re-pinned to the front by normalize, and that order then
  appears everywhere an agent list is shown — the default/tool-agent dropdowns, discussion
  participants, and consensus voters — while the user-authored degradation chain keeps its own
  sequence (AC-R20). Back-compat: an existing config with no `order_seq` keeps its current visual
  order until the first drag.
- **Pick an icon:** Given the settings view, When the user opens an agent row's emoji picker and
  clicks an emoji, Then it is written into that agent's `icon` text field and saved like any manual
  edit; the manual text input still accepts free entry. The picker is a **display-only input
  affordance** in the web console — it writes back to the same `icon` field and changes no
  persistence or normalization (AC-R11 still governs trim / empty / 16-char cap).

## Domain events (wire)

Consumes `get_settings`, `save_settings`, `load_workspace_setting`, `save_workspace_setting`. Emits `settings`, `workspace_setting`. See the
[shared protocol](../../../shared/api-conventions/websocket-protocol.md).

## Interactions

- **agent-session** — receives the resolved `envOverrides` / `model` for each run via
  `resolveSessionLaunch(sessionId)`.
- **session-registry** — supplies the active `sessionId` that the binding is keyed on (the
  binding itself lives in agent-config's own `~/.c3/state.json`, not the registry's state).
- **web-console** — renders the full-page settings view and sends `get_settings` / `save_settings`.
- **discussion** — reads `maxRoundsPerStage` via `getMaxRoundsPerStage()` as the per-stage round
  cap for the orchestration loop (AC-R9).

## Data dictionary

- **Agent** — a Claude Code launch profile keyed by `id`.
- **Default agent** — the agent unbound sessions launch with.
- **settings.json** — the agent registry at `~/.c3/settings.json`.
- **state.json (`~/.c3`)** — the two-key session→agent binding (v2): `pendingIntents` (intent) +
  `sessionAgents` (facts with frozen vendor); distinct from the session-registry's
  `~/.claude/c3/state.json`.
- **Pending intent** — the mutable desired-agent entry for a not-yet-run session (`pendingIntents`).
- **Session fact** — the settled agent + frozen vendor for a real session (`sessionAgents`).
- **Vendor ownership** — the immutable invariant that a session's vendor is fixed at first bind.
