# Glossary

Business and technical terms used across c3 specs. Defined once here; domain docs
reference these rather than redefining them.

| Term                                                | Definition                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| --------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **c3**                                              | Claude Code Center. The application: a local web UI that gates Claude Code's tool use through a browser.                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **Agent run**                                       | A single `query()` invocation driven by one user prompt. Streams assistant text, tool activity, and permission requests until it completes or errors.                                                                                                                                                                                                                                                                                                                                                                                                               |
| **Permission request**                              | A server→client event asking the user to authorize a specific tool call (tool name + input). Emitted from the SDK `canUseTool` callback. Identified by a `requestId`.                                                                                                                                                                                                                                                                                                                                                                                               |
| **Permission decision**                             | The user's answer to a permission request: `allow` or `deny`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **Permission mode**                                 | The policy the SDK uses to decide which tools need a request. One of `default`, `auto`, `plan`, `acceptEdits`, `bypassPermissions`. Persists for the life of a connection.                                                                                                                                                                                                                                                                                                                                                                                          |
| **canUseTool**                                      | The SDK callback c3 supplies. Invoked for sensitive tool calls; c3 routes the question to the browser and returns the resolved decision.                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **Sensitive tool**                                  | A tool the SDK classifies as needing approval under the active mode (e.g. `Write`, `Edit`, dangerous `Bash`). Read-only tools are auto-allowed by the SDK and never produce a request.                                                                                                                                                                                                                                                                                                                                                                              |
| **Auto-deny**                                       | The outcome when a pending permission request is cleared because the run was stopped (`stop_run` / delete / workspace removal) before the user decided. There is no timeout-based auto-deny — an unanswered request blocks indefinitely, and switching the view never denies it.                                                                                                                                                                                                                                                                                    |
| **Wire protocol**                                   | The JSON message contract over the `/ws` WebSocket. Two discriminated unions: `ClientToServer` and `ServerToClient`, defined in `shared/src/protocol.ts`.                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Project directory**                               | The seed workspace directory passed as `--project` (defaults to the current working directory), used as the SDK `cwd`. Claude reads and writes files relative to it.                                                                                                                                                                                                                                                                                                                                                                                                |
| **Session**                                         | A Claude conversation in a workspace. Its execution is owned by a process-wide **Session Runtime**; a connection only **views** one session at a time. Runs are not bound to a connection (ADR 0006).                                                                                                                                                                                                                                                                                                                                                               |
| **Session Runtime**                                 | The process-wide owner of one session's run (`server/src/runs.ts`): its abort/handle, a `baseline + buffer` of wire events for replay, the current viewers, and live status. Shared across connections; survives switching/disconnect.                                                                                                                                                                                                                                                                                                                              |
| **Session status**                                  | A session's live run state: `idle`, `running`, `awaiting_permission`, or `team`. Broadcast to all connections via `session_status` to drive sidebar badges.                                                                                                                                                                                                                                                                                                                                                                                                         |
| **Viewed session**                                  | The single session a connection currently watches. The next `user_prompt` from that connection runs against it; switching the view never stops the previous session's run.                                                                                                                                                                                                                                                                                                                                                                                          |
| **Run handle**                                      | Live controls for an in-flight run handed to the WS handler (`setPermissionMode`), letting a mode change apply mid-run.                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Interrupt / abort**                               | Stopping an in-flight run via the SDK `query.interrupt()`. Triggered only by `stop_run`, `delete_session`, or `remove_workspace` — never by switching the viewed session or closing the connection.                                                                                                                                                                                                                                                                                                                                                                 |
| **Static embed**                                    | The web bundle inlined into the compiled binary (`server/src/static-embed.ts`, generated and gitignored). Served when no filesystem `web/dist` is present.                                                                                                                                                                                                                                                                                                                                                                                                          |
| **claude CLI**                                      | The host-installed `claude` executable the SDK spawns to run the agent. Resolved from `$CLAUDE_PATH` or the PATH. A hard runtime dependency.                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **turn_end**                                        | The terminal server→client event for one prompt→result turn, with `reason` `complete` (incl. a stopped run) or `error`. Never means the session ended — it stays alive for the next prompt.                                                                                                                                                                                                                                                                                                                                                                         |
| **Requirement (需求)**                              | A project-scoped ledger item — `title`, `content`, `priority` (P0–P3), `status`, optional intra-project dependencies — persisted in `~/.c3/c3.db`, keyed by the resolved workspace path. The unit the requirement list, refine, and launch-development all operate on (ADR 0007).                                                                                                                                                                                                                                                                                   |
| **需求级别 (Requirement priority)**                 | A requirement's priority, one of `P0`, `P1`, `P2`, `P3` (P0 highest).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| **模块名称 (Module)**                               | A requirement's owning module — a free-text label the communication agent infers from the item's title/content (e.g. 认证、会话、需求管理). Persisted as `requirements.module` (`TEXT NOT NULL DEFAULT ''`); `''` when unidentified or for historical rows. Lays the data groundwork for later module-based organization/display (ADR 0007).                                                                                                                                                                                                                        |
| **New Requirement session (Communication session)** | The per-project, long-lived, read-only agent session used to refine ideas into requirement items. A real SDK session kept out of the normal session list (the hidden set); runs in forced `default` mode and cannot edit/write/run commands/spawn sub-agents/run slash commands (ADR 0007).                                                                                                                                                                                                                                                                         |
| **save_requirements**                               | The in-process MCP tool (`mcp__c3__save_requirements`) the communication agent calls to propose persisting a batch of requirements. It reuses the permission gateway: c3 pops a confirmation; the tool handler writes to the ledger only after the user allows (ADR 0007).                                                                                                                                                                                                                                                                                          |
| **启动开发 (Launch development)**                   | The action that turns a `todo` requirement into a **background normal session** running the configurable development skill (`devSkill` in system settings; empty by default ⇒ no skill prefix) with the requirement content, sets the requirement to `in_progress`, and records `lastDevSessionId` for the development back-link. The run never auto-completes the requirement.                                                                                                                                                                                     |
| **自动提交 lint 自愈 (Auto-commit lint self-heal)** | When the automation orchestrator's `done` auto-commit is rejected by a **pre-commit lint hook** (`commitAndPush` returns `failure: 'commit-hook'`), it self-heals instead of hard-stopping by handing the failure to the **dev agent once** — lint toolchains differ per project, so there is no portable fix command. It resumes the same dev session with the lint error summary, lets the agent fix it, then retries the commit **once**. Any non-lint commit/push failure, or a lint failure surviving the single agent attempt, is a hard stop (RM-A13/RM-A6). |
| **c3.db**                                           | The SQLite requirement ledger at `~/.c3/c3.db`. Distinct from the registry's `state.json` (under `~/.claude/c3/`); accessed through a cross-runtime driver adapter (`node:sqlite` / `bun:sqlite`) that fails soft so c3 still boots without it (ADR 0007).                                                                                                                                                                                                                                                                                                          |
| **Streaming input**                                 | The SDK mode where `query()`'s prompt is a controlled async-iterable (`InputStream`) rather than a one-shot string. Pushing a turn appends to the same live session; closing the stream ends the query. c3 runs **all** sessions this way (ADR 0008): it keeps the lead process alive for agent teams, makes `setPermissionMode`/`interrupt` effective, and reproduces one-shot behaviour for normal sessions by closing the stream on `result`.                                                                                                                    |
| **Agent team**                                      | A Claude Code SDK arrangement where a **team lead** delegates work to **teammates**. c3 detects it at runtime when a team tool (`TeamCreate`, `SendMessage`, or a background `Agent`) is used, upgrading the session to a **persistent session** (AS-R14, ADR 0008).                                                                                                                                                                                                                                                                                                |
| **Team lead**                                       | The primary agent of an agent team — the `query()` run c3 owns. In a team session its process stays alive between turns (it does not exit on `result`) to coordinate teammates; c3 keeps the run in flight until the user explicitly stops it.                                                                                                                                                                                                                                                                                                                      |
| **Teammate**                                        | A delegated agent spawned by the team lead, e.g. a background `Agent` (`run_in_background: true`) that runs detached and reports its result back to the lead asynchronously. Killing the lead process would orphan teammates — the reason the lead must stay alive (ADR 0008).                                                                                                                                                                                                                                                                                      |
| **Persistent session**                              | A session whose run stays in flight between turns rather than ending each turn. Currently only a `team` session: the lead process is alive between turns, status holds at `team` (not `idle`), and it ends only on explicit user stop (`stop_run` / `delete_session` / `remove_workspace`).                                                                                                                                                                                                                                                                         |
| **team_upgraded**                                   | The one-shot server→client event (and `SessionStatus` `team`) marking a session as a persistent agent team. Recorded in the session buffer so reconnect replay shows it; drives the team badge and the always-enabled composer in the UI.                                                                                                                                                                                                                                                                                                                           |
| **Vendor adapter**                                  | One agent vendor's implementation of the neutral abstraction (`adapters/<vendor>/`): its `AgentDriver`, `ApprovalBridge`, `SessionStore`, and `AdapterCapabilities`. c3 selects one by `VendorId` (`claude`/`codex`/`opencode`) and drives it through the neutral faces only. Today only the Claude reference adapter exists (ADR-0011).                                                                                                                                                                                                                            |
| **AgentDriver**                                     | The neutral lifecycle + streaming face of a vendor: `start(opts)` returns an `AgentRun` exposing `sessionId()`, `messages()` (a canonical stream), and `abort()` (required), plus capability-gated optional controls. Generalizes the Claude `query()` loop (ADR-0011).                                                                                                                                                                                                                                                                                             |
| **ApprovalBridge**                                  | The neutral approval channel — intercept → suspend → write back — via `onRequest(handler) → disposer`. For vendors with `perToolApproval` the handler fires per tool (Claude resolves the blocking `canUseTool`; OpenCode POSTs a REST decision); Codex has no live point and degrades to launch-time policy (ADR-0011).                                                                                                                                                                                                                                            |
| **SessionStore**                                    | The neutral session-history face that hides the dirtiest coupling (Claude's direct JSONL reads, OpenCode REST, Codex thread items) behind `list`/`read`, returning canonical `CanonicalMessage[]`; optional `rename`/`delete` (ADR-0011).                                                                                                                                                                                                                                                                                                                           |
| **AdapterCapabilities**                             | The probed ledger of a vendor's optional/degradable abilities — exactly six boolean flags: `interrupt`, `setActionMode`, `streamingPush`, `inProcessMcp`, `forkSession`, `perToolApproval`. Required capabilities carry no flag. The upper layer probes a flag (and method presence) before reaching for a divergent control, degrading when absent (ADR-0011).                                                                                                                                                                                                     |
| **perToolApproval**                                 | The capability flag for in-the-loop, per-tool runtime approval. `true` for Claude and OpenCode; **`false` for Codex** (008: stdin closes after dispatch, no write-back half-channel) — the addition beyond Claude's five proprietary controls (ADR-0011).                                                                                                                                                                                                                                                                                                           |
| **Canonical message**                               | The vendor-spanning message envelope (ADR-0011, per the 010 diff): `vendor` tag + `sessionId` (unconditional) + discounted `role`/`blocks`/`ts`/`turnId?` + two-level `vendorExtra` overflow. Blocks are append-with-upsert; a tool's return is **embedded** on its `tool_use` block (`result?`), not a standalone block.                                                                                                                                                                                                                                           |
| **ActionMode × ToolGate (neutral permission grid)** | The 2-axis replacement for Claude's five-way `PermissionMode` (ADR-0011): `ActionMode{plan,build}` (what the run may do) × `ToolGate{always-ask, on-sensitive, trusted-prefix, never-ask}` (how tools are gated). Each adapter translates its native mode(s) in; the grid does not round-trip 1:1 (Claude `auto`'s bias and `always-ask` have no exact peer). The neutral policy is `(toolName, input, ctx) → allow\|ask\|deny`.                                                                                                                                    |
| **release (build)**                                 | The thin orchestration layer over the existing build/binary primitives that produces multi-platform artifacts (`pnpm release:build`). It does not replace `pnpm build` or `pnpm binary`; it sequences three race-free phases — Phase0 web build (once) → Phase1 generate-static-embed (once) → Phase2 parallel `bun --compile` fan-out (read-only). See `specs/non-functional/release.md`, ADR-0010.                                                                                                                                                                |
| **channel**                                         | A named distribution track for release artifacts (e.g. `stable`, `nightly`). Determines artifact naming/suffix and which version is published. **Placeholder** in release 1/7 — the orchestrator builds without a channel concept yet.                                                                                                                                                                                                                                                                                                                              |
| **hardened (tier)**                                 | A build-hardening level passed to `buildTarget({ harden })` — the ladder from `default` up through stripped symbols, reproducible builds, signing, and notarization. **Placeholder** in release 1/7: only the `default` tier is honored; other values warn and fall back.                                                                                                                                                                                                                                                                                           |
| **version SoT**                                     | The single source of truth for the artifact version and its build-time injection (currently the package manifest; `--version` → `0.1.0`). A unified version-SoT + git-tag/channel injection is a **placeholder** for a later release wave.                                                                                                                                                                                                                                                                                                                          |
