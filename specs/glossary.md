# Glossary

Business and technical terms used across c3 specs. Defined once here; domain docs
reference these rather than redefining them.

| Term                                         | Definition                                                                                                                                                                                                                                                                                                                                   |
| -------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **c3**                                       | Claude Code Center. The application: a local web UI that gates Claude Code's tool use through a browser.                                                                                                                                                                                                                                     |
| **Agent run**                                | A single `query()` invocation driven by one user prompt. Streams assistant text, tool activity, and permission requests until it completes or errors.                                                                                                                                                                                        |
| **Permission request**                       | A server→client event asking the user to authorize a specific tool call (tool name + input). Emitted from the SDK `canUseTool` callback. Identified by a `requestId`.                                                                                                                                                                        |
| **Permission decision**                      | The user's answer to a permission request: `allow` or `deny`.                                                                                                                                                                                                                                                                                |
| **Permission mode**                          | The policy the SDK uses to decide which tools need a request. One of `default`, `auto`, `plan`, `acceptEdits`, `bypassPermissions`. Persists for the life of a connection.                                                                                                                                                                   |
| **canUseTool**                               | The SDK callback c3 supplies. Invoked for sensitive tool calls; c3 routes the question to the browser and returns the resolved decision.                                                                                                                                                                                                     |
| **Sensitive tool**                           | A tool the SDK classifies as needing approval under the active mode (e.g. `Write`, `Edit`, dangerous `Bash`). Read-only tools are auto-allowed by the SDK and never produce a request.                                                                                                                                                       |
| **Auto-deny**                                | The outcome when a pending permission request is cleared because the run was stopped (`stop_run` / delete / workspace removal) before the user decided. There is no timeout-based auto-deny — an unanswered request blocks indefinitely, and switching the view never denies it.                                                             |
| **Wire protocol**                            | The JSON message contract over the `/ws` WebSocket. Two discriminated unions: `ClientToServer` and `ServerToClient`, defined in `shared/src/protocol.ts`.                                                                                                                                                                                    |
| **Project directory**                        | The seed workspace directory passed as `--project` (defaults to the current working directory), used as the SDK `cwd`. Claude reads and writes files relative to it.                                                                                                                                                                         |
| **Session**                                  | A Claude conversation in a workspace. Its execution is owned by a process-wide **Session Runtime**; a connection only **views** one session at a time. Runs are not bound to a connection (ADR 0006).                                                                                                                                        |
| **Session Runtime**                          | The process-wide owner of one session's run (`server/src/runs.ts`): its abort/handle, a `baseline + buffer` of wire events for replay, the current viewers, and live status. Shared across connections; survives switching/disconnect.                                                                                                       |
| **Session status**                           | A session's live run state: `idle`, `running`, or `awaiting_permission`. Broadcast to all connections via `session_status` to drive sidebar badges.                                                                                                                                                                                          |
| **Viewed session**                           | The single session a connection currently watches. The next `user_prompt` from that connection runs against it; switching the view never stops the previous session's run.                                                                                                                                                                   |
| **Run handle**                               | Live controls for an in-flight run handed to the WS handler (`setPermissionMode`), letting a mode change apply mid-run.                                                                                                                                                                                                                      |
| **Interrupt / abort**                        | Stopping an in-flight run via the SDK `query.interrupt()`. Triggered only by `stop_run`, `delete_session`, or `remove_workspace` — never by switching the viewed session or closing the connection.                                                                                                                                          |
| **Static embed**                             | The web bundle inlined into the compiled binary (`server/src/static-embed.ts`, generated and gitignored). Served when no filesystem `web/dist` is present.                                                                                                                                                                                   |
| **claude CLI**                               | The host-installed `claude` executable the SDK spawns to run the agent. Resolved from `$CLAUDE_PATH` or the PATH. A hard runtime dependency.                                                                                                                                                                                                 |
| **turn_end**                                 | The terminal server→client event for one prompt→result turn, with `reason` `complete` (incl. a stopped run) or `error`. Never means the session ended — it stays alive for the next prompt.                                                                                                                                                  |
| **Requirement (需求)**                       | A project-scoped ledger item — `title`, `content`, `priority` (P0–P3), `status`, optional intra-project dependencies — persisted in `~/.c3/c3.db`, keyed by the resolved workspace path. The unit the requirement list, refine, and launch-development all operate on (ADR 0007).                                                            |
| **需求级别 (Requirement priority)**          | A requirement's priority, one of `P0`, `P1`, `P2`, `P3` (P0 highest).                                                                                                                                                                                                                                                                        |
| **模块名称 (Module)**                        | A requirement's owning module — a free-text label the communication agent infers from the item's title/content (e.g. 认证、会话、需求管理). Persisted as `requirements.module` (`TEXT NOT NULL DEFAULT ''`); `''` when unidentified or for historical rows. Lays the data groundwork for later module-based organization/display (ADR 0007). |
| **需求沟通 session (Communication session)** | The per-project, long-lived, read-only agent session used to refine ideas into requirement items. A real SDK session kept out of the normal session list (the hidden set); runs in forced `default` mode and cannot edit/write/run commands/spawn sub-agents/run slash commands (ADR 0007).                                                  |
| **save_requirements**                        | The in-process MCP tool (`mcp__c3__save_requirements`) the communication agent calls to propose persisting a batch of requirements. It reuses the permission gateway: c3 pops a confirmation; the tool handler writes to the ledger only after the user allows (ADR 0007).                                                                   |
| **启动开发 (Launch development)**            | The action that turns a `todo` requirement into a **background normal session** running `/sdd-lite` with the requirement content, sets the requirement to `in_progress`, and records `lastDevSessionId` for the development back-link. The run never auto-completes the requirement.                                                         |
| **c3.db**                                    | The SQLite requirement ledger at `~/.c3/c3.db`. Distinct from the registry's `state.json` (under `~/.claude/c3/`); accessed through a cross-runtime driver adapter (`node:sqlite` / `bun:sqlite`) that fails soft so c3 still boots without it (ADR 0007).                                                                                   |
