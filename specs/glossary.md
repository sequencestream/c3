# Glossary

Business and technical terms used across c3 specs. Defined once here; domain docs
reference these rather than redefining them.

- **c3**: Code Creative Center. The application: a local web UI that gates Claude Code's tool use through a browser.
- **Agent run**: A single `query()` invocation driven by one user prompt. Streams assistant text, tool activity, and permission requests until it completes or errors.
- **VendorModeCatalog**: The per-vendor declaration of native mode tokens (2026-06-07-012). Each entry pairs a vendor's `token` (a `ModeToken`) with its i18n `labelCode` and the neutral `ActionMode × ToolGate` grid cell it maps to. Declared in `adapters/<vendor>/modes.ts`; aggregated in `MODE_CATALOGS` (`adapters/index.ts`, `Record<VendorId, VendorModeCatalog>`) — the compile-time exhaustiveness pin. Shipped to the web on `settings.vendorModes` so the console renders the mode picker by `vendor` with zero `if (vendor === …)`.
- **Permission decision**: The user's answer to a permission request: `allow` or `deny`.
- **Permission mode**: The policy applied to tool calls in the viewed session. As of 2026-06-07-012 it is a vendor-native {@link ModeToken} carried on the wire as `SessionInfo.mode` / `set_mode.mode` / `mode_changed.mode`, interpreted through the session's vendor `VendorModeCatalog` as a neutral `ActionMode × ToolGate` grid cell. Claude's former five-value `PermissionMode` union remains as that vendor's token set; the generic type is `ModeToken` (a `string`). Persists per-session, seeded from the project's `defaultMode`. See also {@link VendorModeCatalog}.
- **canUseTool**: The SDK callback c3 supplies. Invoked for sensitive tool calls; c3 routes the question to the browser and returns the resolved decision.
- **Sensitive tool**: A tool the SDK classifies as needing approval under the active mode (e.g. `Write`, `Edit`, dangerous `Bash`). Read-only tools are auto-allowed by the SDK and never produce a request.
- **Auto-deny**: The outcome when a pending permission request is cleared because the run was stopped (`stop_run` / delete / workspace removal) before the user decided. There is no timeout-based auto-deny — an unanswered request blocks indefinitely, and switching the view never denies it.
- **Wire protocol**: The JSON message contract over the `/ws` WebSocket. Two discriminated unions: `ClientToServer` and `ServerToClient`, defined in `shared/src/protocol.ts`.
- **Authentication (认证)**: Establishing **who** a connection is before it may drive agents. The mandatory precondition for binding the server to any non-loopback address (constitution C-SEC-5, ADR-0023). As of 2026-06-11 it is **contract-only**: types, persisted config shape, and wire messages exist; no runtime (middleware/login/hashing/token signing) and no actual relaxation of the localhost-only bind yet. See the [auth domain](domains/core/auth/auth-overview.md).
- **AuthProvider**: The `kind`-discriminated authentication backend abstraction (ADR-0023, `shared/src/protocol.ts`). `kind: 'basic'` (**BasicAuthProvider** — `accounts: { username, PHC passwordHash }[]` + `adminUsername`: **multiple accounts, exactly one admin**) is runtime-live; every account may sign in, `adminUsername` designates the single config-authority account (no RBAC). `kind: 'oauth'` (**OAuthAuthProvider** — generic OIDC: `issuer`, `clientId`, `clientSecretRef`, `redirectUri`, `scopes`, `usePkce`, `allowedEmails`, `adminEmail`) is **contract-only** (config persists; `/auth/callback`, discovery, PKCE, token exchange, JWKS, session minting all deferred to the OAuth-runtime task, so enabling auth still works only with `basic`); `adminEmail` is the single admin email and must be a member of `allowedEmails`. `basic` and `oauth` are mutually exclusive (one `provider` union). `sso`/multi-tenant remain the reserved extension point (add a `kind` arm + a server zod arm; nothing else changes). The same neutral-abstraction + per-kind-extension shape as the ADR-0011 vendor model.
- **AuthConfig**: The authentication configuration hung on `SystemSettings.auth?` — `{ enabled, provider, session, exposure? }` (ADR-0023). Absent block or `enabled: false` ⇒ no auth (the C-SEC-5 default); a malformed block normalizes (fail-soft) to disabled. `session` is the provider-neutral `AuthSessionPolicy` (token `ttlSeconds` + `signingKeyRef`, a reference to the signing key, never the key itself); `exposure.bindAddress` records network-exposure intent.
- **Session token (会话令牌)**: The provider-neutral issued credential a client presents after login — `AuthSessionToken { tokenId, subject, issuedAt, expiresAt }` (ADR-0023). Minted on successful `login`, carried opaquely on the wire, verified per request (signing/verification deferred to a later task). Distinct from a c3 **Session** (an agent conversation) and from the session-registry's persisted state.
- **passwordHash / signingKeyRef / clientSecretRef**: The "secret-by-indirection" fields of `AuthConfig` (ADR-0023, AUTH-R3/R4/R9). `passwordHash` stores a password only as a PHC hash — never plaintext; the plaintext `AuthLoginRequest.password` exists in transit only. `signingKeyRef` is a _reference_ (env var name / keystore id) to the token signing secret, which is never persisted in `settings.json`. `clientSecretRef` (on `OAuthAuthProvider`) follows the same discipline: a reference to the OAuth client secret, never the secret value itself; zod strips any stray plaintext `clientSecret` a client round-trips.
- **Workspace directory**: The seed workspace directory passed as `--workspace` (defaults to the current working directory), used as the SDK `cwd`. Claude reads and writes files relative to it. The legacy flag `--project` remains as a deprecated alias for one cycle.
- **Session**: A Claude conversation in a workspace. Its execution is owned by a process-wide **Session Runtime**; a connection only **views** one session at a time. Runs are not bound to a connection (ADR 0006).
- **Works**: The top-level navigation tab that groups a user's development work sessions. Formerly named "Sessions" (renamed 2026-06-08 to disambiguate from the general session concept). Renders as the `console` tab internally. See also **Session**.
- **Work session**: A session displayed under the Works tab — the primary development conversation. Distinguishes a Works-tab session from intent sessions, discussion sessions, and schedule sessions.
- **Session Runtime**: The process-wide owner of one session's run (`server/src/runs.ts`): its abort/handle, a `baseline + buffer` of wire events for replay, the current viewers, and live status. Shared across connections; survives switching/disconnect.
- **RunKind**: The single source-of-truth taxonomy (`shared/src/protocol.ts`) for a run/agent invocation's origin: `session` (general dev session — user console, intent→dev hand-off, dev-turn), `intent` (read-only intent-communication session), `discussion` (orchestrator + research), `schedule` (a scheduler-launched, socket-less run), `consensus` (a consensus vote), `tool` (internal tool call — completion judging + title derivation). Replaces the old two-value `SessionKind` (`'normal' | 'intent'`; `'normal' → 'session'`). Carried on `SessionRuntime.kind` and the `run:started`/`run:settled` events. **`schedule` is a trigger source, not a run type:** a schedule _triggered_ by a user session reacts to a `session`-kind run; `schedule` only tags the scheduler's own socket-less run. Event-triggered schedules fire only on `kind === 'session'`.
- **Session status**: A session's live run state: `idle`, `running`, `awaiting_permission`, or `team`. Broadcast to all connections via `session_status` to drive sidebar badges.
- **Viewed session**: The single session a connection currently watches. The next `user_prompt` from that connection runs against it; switching the view never stops the previous session's run.
- **Run handle**: Live controls for an in-flight run handed to the WS handler (`setPermissionMode`), letting a mode change apply mid-run.
- **Interrupt / abort**: Stopping an in-flight run via the SDK `query.interrupt()`. Triggered only by `stop_run`, `delete_session`, or `remove_workspace` — never by switching the viewed session or closing the connection.
- **Static embed**: The web bundle inlined into the compiled binary (`server/src/static-embed.ts`, generated and gitignored). Served when no filesystem `web/dist` is present.
- **claude CLI**: The host-installed `claude` executable the SDK spawns to run the agent. Resolved from `$CLAUDE_PATH` or the PATH (via the ProcessLauncher). A hard runtime dependency — the Claude instance of the per-vendor **host binary** intent (ADR-0012).
- **turn_end**: The terminal server→client event for one prompt→result turn, with `reason` `complete` (incl. a stopped run) or `error`. Never means the session ended — it stays alive for the next prompt.
- **Intent (意图)**: A project-scoped ledger item — `title`, `content`, `priority` (P0–P3), `status`, optional intra-project dependencies — persisted in `~/.c3/c3.db`, keyed by the resolved workspace path. The unit the intent list, refine, and launch-development all operate on (ADR 0007). Renamed from **Requirement** (requirements→intents, PR-2). **Disambiguation:** the domain **Intent** (capitalized; a persisted work-unit entity) is NOT the lowercase word _intent_ used elsewhere for unrelated concepts — a Vue child component's event _intent_ (a user-action emit), the ADR-0011 permission _intent_, the consensus vendor-free _intent_, and the orchestrator's next-subtopic / broadcast _intent_. Those are same-word / different-context (cross-referenced here) and are deliberately untouched by this rename.
- **意图级别 (Intent priority)**: An intent's priority, one of `P0`, `P1`, `P2`, `P3` (P0 highest).
- **模块名称 (Module)**: An intent's owning module — a free-text label the communication agent infers from the item's title/content (e.g. 认证、会话、意图管理). Persisted as `intents.module` (`TEXT NOT NULL DEFAULT ''`); `''` when unidentified or for historical rows. Lays the data groundwork for later module-based organization/display (ADR 0007).
- **New Intent session (Communication session)**: The per-project, long-lived, read-only agent session used to refine ideas into intent items. A real SDK session kept out of the normal session list (the hidden set); runs in forced `default` mode and cannot edit/write/run commands/spawn sub-agents/run slash commands (ADR 0007).
- **save_intents**: The in-process MCP tool (`mcp__c3__save_intents`) the communication agent calls to propose persisting a batch of intents. It reuses the permission gateway: c3 pops a confirmation; the tool handler writes to the ledger only after the user allows (ADR 0007).
- **启动开发 (Launch development)**: The action that turns a `todo` intent into a **background normal session** running the configurable development skill (`devSkill` in system settings; empty by default ⇒ no skill prefix) with the intent content, sets the intent to `in_progress`, and records `lastDevSessionId` for the development back-link. The run never auto-completes the intent.
- **自动提交 lint 自愈 (Auto-commit lint self-heal)**: When the automation orchestrator's `done` auto-commit is rejected by a **pre-commit lint hook** (`commitAndPush` returns `failure: 'commit-hook'`), it self-heals instead of hard-stopping by handing the failure to the **dev agent once** — lint toolchains differ per project, so there is no portable fix command. It resumes the same dev session with the lint error summary, lets the agent fix it, then retries the commit **once**. Any non-lint commit/push failure, or a lint failure surviving the single agent attempt, is a hard stop (RM-A13/RM-A6).
- **c3.db**: The SQLite intent ledger at `~/.c3/c3.db`. Distinct from the registry's `state.json` (under `~/.claude/c3/`); accessed through a cross-runtime driver adapter (`node:sqlite` / `bun:sqlite`) that fails soft so c3 still boots without it (ADR 0007).
- **Streaming input**: The SDK mode where `query()`'s prompt is a controlled async-iterable (`InputStream`) rather than a one-shot string. Pushing a turn appends to the same live session; closing the stream ends the query. c3 runs **all** sessions this way (ADR 0008): it keeps the lead process alive for agent teams, makes `setPermissionMode`/`interrupt` effective, and reproduces one-shot behaviour for normal sessions by closing the stream on `result`.
- **Agent team**: A Claude Code SDK arrangement where a **team lead** delegates work to **teammates**. c3 detects it at runtime when a team tool (`TeamCreate`, `SendMessage`, or a background `Agent`) is used, upgrading the session to a **persistent session** (AS-R14, ADR 0008). **Claude-locked** (AS-R21, 2026-06-06-006): a session becomes a team only when its vendor has `streamingPush` (a resident lead running in-process `TeamCreate`/`SendMessage`) — only Claude does; a non-Claude session is never marked `team`. A heterogeneous teammate is a deferred capability.
- **Team lead**: The primary agent of an agent team — the `query()` run c3 owns. In a team session its process stays alive between turns (it does not exit on `result`) to coordinate teammates; c3 keeps the run in flight until the user explicitly stops it.
- **Teammate**: A delegated agent spawned by the team lead, e.g. a background `Agent` (`run_in_background: true`) that runs detached and reports its result back to the lead asynchronously. Killing the lead process would orphan teammates — the reason the lead must stay alive (ADR 0008).
- **Persistent session**: A session whose run stays in flight between turns rather than ending each turn. Currently only a `team` session: the lead process is alive between turns, status holds at `team` (not `idle`), and it ends only on explicit user stop (`stop_run` / `delete_session` / `remove_workspace`).
- **team_upgraded**: The one-shot server→client event (and `SessionStatus` `team`) marking a session as a persistent agent team. Recorded in the session buffer so reconnect replay shows it; drives the team badge and the always-enabled composer in the UI.
- **AgentDriver**: The neutral lifecycle + streaming face of a vendor: `start(opts)` returns an `AgentRun` exposing `sessionId()`, `messages()` (a canonical stream), and `abort()` (required), plus capability-gated optional controls. Generalizes the Claude `query()` loop (ADR-0011).
- **AdapterCapabilities**: The probed ledger of a vendor's optional/degradable abilities — two parts (ADR-0011, amended 2026-06-07): **(a)** exactly seven **boolean** live-run flags: `interrupt`, `setActionMode`, `streamingPush`, `inProcessMcp`, `forkSession`, `perToolApproval`, `taskStore`; **(b)** a structured sub-ledger **`sessions: SessionCapabilities`** for the session-lifecycle operations (`list` / `read` / `resume` / `rename` / `delete`), each a 4-state `CapabilityState` (`'none'` / `'partial'` / `'full'` / `'temporarily-unavailable'`) — a boolean could not tell `none` (structural NO) apart from `temporarily-unavailable` (mechanism exists, not currently reachable), and that distinction is what the UI renders. Required capabilities carry no flag. The upper layer probes a flag (and method presence) before reaching for a divergent control, degrading when absent; the session sub-ledger's states drive row-action gating by state, with **zero `if (vendor === …)`** (ADR-0011).
- **Canonical message**: The vendor-spanning message envelope (ADR-0011, per the 010 diff): `vendor` tag + `sessionId` (unconditional) + discounted `role`/`blocks`/`ts`/`turnId?` + two-level `vendorExtra` overflow. Blocks are append-with-upsert; a tool's return is **embedded** on its `tool_use` block (`result?`), not a standalone block. **Promoted to the wire** (`shared/protocol.ts`, SDK-free) in ADR-0013 — the kernel re-exports it; the wire gains only a `vendor` dimension, never a per-vendor schema.
- **ActionMode × ToolGate (neutral permission grid)**: The 2-axis replacement for Claude's five-way `PermissionMode` (ADR-0011): `ActionMode{plan,build}` (what the run may do) × `ToolGate{always-ask, on-sensitive, trusted-prefix, never-ask}` (how tools are gated). Each adapter declares a {@link VendorModeCatalog} that translates its native mode tokens into this grid — the forward direction via `tokenToGrid`, the reverse (lossy) via `gridToToken`. The neutral policy is `(toolName, input, ctx) → allow|ask|deny`. Promoted to `shared/protocol.ts` in 2026-06-07-012 as the single SoT.
- **ProcessLauncher**: The vendor-agnostic host-binary probe (`server/src/kernel/agent/process/launcher.ts`): `resolve(vendor) → absolute path | null` (precedence `$<PATH_ENV>` → PATH probe → cache), a `HOST_BINARIES` table (binary name / `*_PATH` override / install hint per vendor), and `probe`/`probeAll` for the boot health check. The **first capability gate** — a `null` short-circuits before any adapter or capability (ADR-0012).
- **Adapter registry**: `server/src/kernel/agent/adapters/registry.ts`: maps each implemented vendor to its factory (`VENDOR_FACTORIES`, `{claude}` today) and turns the ProcessLauncher probe into `resolveAvailableAdapters() → { available, missing }`. Probes **before** constructing each adapter; an unresolved binary lands in `missing` (with install hint) and is never built. `available` is the kernel's source of truth for available agent types; `logHostBinaryHealth()` logs it at boot (ADR-0012).
- **AdapterCapability (enum)**: The wire-facing capability enum (`shared/protocol.ts`, ADR-0013) naming the seven optional/degradable capabilities: `interrupt`/`setActionMode`/`streamingPush`/`inProcessMcp`/`forkSession`/`perToolApproval`/`taskStore`. The kernel's `AdapterCapabilities` boolean ledger is keyed by exactly these names; a compile-time assertion pins the two so they cannot drift.
- **Block upsert (two-form)**: The rule that lets both vendor message forms share one model (ADR-0013): blocks are keyed by `(sessionId, block.id)` and **upserted**, not append-only. Claude's whole-message frame and Codex's incremental `ItemUpdated` both collapse to "revise in place, don't stack"; a tool's return back-fills its `tool_use` monotonically. Implemented by `CanonicalAccumulator` / `upsertBlock` (`adapters/canonical-accumulator.ts`).
- **c3SessionId**: The opaque, vendor-free session handle (`"c3s_" + sha256(vendor \0 vendorSessionId)[:32]`, ADR-0013) — the only session id that crosses out of the kernel. Deterministic (no persistence needed) and containing neither the vendor name nor the raw id as a substring, so a vendor id never leaks into a URL or storage key. Resolves to a `SessionRef { vendor, vendorSessionId }` kept inside the kernel.
- **SessionAccessor**: The **read-only** lazy-normalizing union over the available vendors' `SessionStore`s (`kernel/agent/session/accessor.ts`, ADR-0013): `list` merges across vendors (native id hidden in `vendorExtra`), `read` routes to the owning store via a lazily-built `c3 → ref` index. Native stores stay the source of truth — no double-write; the index is a rebuildable runtime cache (存储形态归一、位置不归一).
- **Session→agent binding (two-key)**: The per-session record of which agent a session runs on, split into two key spaces (ADR-0015, agent-config AC-R16): a mutable **pending intent** and a settled **session fact**. Persisted at `~/.c3/state.json` v2 (`pendingIntents` + `sessionAgents`); `getSessionAgentId` reads both. Distinct from `c3SessionId` / `SessionAccessor`, which are about reading transcripts.
- **Pending intent**: The mutable half of the binding (ADR-0015): `pendingId → { agentId, createdAt }` — which agent a not-yet-run session wants. Editable/clearable; copied to a fact at first bind and dropped, else reaped by the janitor (`cleanupStalePendingIntents`) after 7 days. Clearing/reaping an intent never touches the fact space, so it can't orphan a fact.
- **Session fact / vendor ownership**: The settled half of the binding (ADR-0015): `realId → { agentId, vendor }` — the agent a real session ran on plus its **frozen vendor**. The vendor is the immutable invariant: a session's transcript lives only in that vendor's native store (c3 never stores session content), so the vendor cannot change. Same-vendor agent swaps are allowed (`changeSessionAgentFact` → `true`); cross-vendor changes are rejected (`false`). Cross-vendor Fork / **replay-seed** stays deferred (ADR-0011).
- **release (build)**: The thin orchestration layer over the existing build/binary primitives that produces multi-platform artifacts (`pnpm release:build`). It does not replace `pnpm build` or `pnpm binary`; it sequences three race-free phases — Phase0 web build (once) → Phase1 generate-static-embed (once) → Phase2 parallel `bun --compile` fan-out (read-only). See `specs/non-functional/release.md`, ADR-0010.
- **channel**: A named distribution track for release artifacts (e.g. `stable`, `nightly`). Determines artifact naming/suffix and which version is published. **Placeholder** in release 1/7 — the orchestrator builds without a channel concept yet.
- **hardened (tier)**: A build-hardening level passed to `buildTarget({ harden })` — the ladder from `default` up through stripped symbols, reproducible builds, signing, and notarization. **Placeholder** in release 1/7: only the `default` tier is honored; other values warn and fall back.
- **version SoT**: The single source of truth for the artifact version and its build-time injection (currently the package manifest; `--version` → `0.1.0`). A unified version-SoT + git-tag/channel injection is a **placeholder** for a later release wave.
